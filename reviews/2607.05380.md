# Review — TabPack: Efficient Hyperparameter Ensembles for Tabular Deep Learning

- **arXiv:** [2607.05380](https://arxiv.org/abs/2607.05380) (v1, cs.LG, Jul 2026)
- **Authors:** Yury Gorishniy, Akim Kotelnikov, Ivan Rubachev, Artem Babenko
- **Venue signal:** styled as ICML submission (Yandex / TabM · FT-Transformer group)

## Problem

Efficient MLP ensembles (TabM) are now a strong, practical tabular-DL family, but
they still inherit tabular ML's chronic pain: **no universally good default
hyperparameters**, so top-tier results require expensive per-dataset tuning. TabM
in particular shares one hyperparameter set across all its base MLPs, so its
diversity comes only from init/dropout/batch order — and it must be tuned.
TabPack asks whether an ensemble can be **strong out-of-the-box with little or no
tuning**.

## Method

TabPack samples many (default **m=64**) MLPs with *different* hyperparameters in a
single run, trains them in parallel, and greedily selects ensemble members on the
fly. Key pieces:

1. **Packed hyperparameter ensembling** (extends Packed-Ensembles). Heterogeneous
   MLPs are padded to the max width/depth and stacked along a "pack" dimension, so
   all base models run in one batched `bmm`; narrower/shallower models are masked
   or skip layers. An analogous **optimizer pack** vectorizes per-member learning
   rates / weight decay. Packing is dramatically faster than sequential or
   multi-process training on one GPU.
2. **Online ensembling.** The ensemble is rebuilt greedily each epoch from a pool
   of *intermediate* base-model checkpoints plus prior members (so the same base
   model at different epochs counts as distinct candidates), with early stopping on
   ensemble validation score (à la TabM). Base models that stall are dropped to
   reallocate compute.
3. **Tuning-free workflow.** The user supplies only sampling *ranges*, not exact
   values. Defaults (fixed width 384, depth ∈[1,3], Muon optimizer, periodic
   feature embeddings `†`) are never tuned per dataset. Every reported number is a
   single run (conservative multi-seed protocol reruns only the selected members).

## Key results

Benchmark = 17 medium-to-large datasets (8 from TabReD temporal splits + 9 others,
10K–13.6M rows, up to 900+ features).

- **TabPack† (default, single run) reaches mean rank 1.6**, on par with
  extensively **tuned** TabM/MLP/RealMLP/ModernNCA/XGBoost — but at a fraction of
  the wall-clock cost (tuning-free).
- **`TabPack†_MacBook`** (Apple M4 Pro, `mps`) finishes a single run faster than
  tuning some baselines on an A100, at competitive accuracy.
- **Online > offline:** beats `TabPack_Offline` and `MLP_HPE` (its multi-process,
  post-hoc-ensemble cousin), and is more stable in the worst-25% of datasets.
- **Scales:** on 5 datasets with ≥1M rows (up to 13.6M) it wins 4/5 vs TabM/MLP/
  XGBoost (TFMs OOM); especially strong on **large regression**. Competitive but
  not dominant on small IID TabArena data.
- Tuning shared HPs (`TabPack_SameHP`) gains only rank 1.4 vs 1.6 but costs 65.5h
  vs 2.4h — so HP diversity's real job is **enabling tuning-free use**, not
  boosting accuracy via diversity per se. Muon consistently beats AdamW; feature
  embeddings matter a lot.

## Fit with the tabular curriculum

A **strong-default trained-DL baseline** in the RealMLP/TabM line that any honest
comparison must now include — squarely on the **honest-baseline** spine
(TabReD → TabArena → BeyondArena), and from the same group that produced TabM and
the numerical-embedding work. Its TabReD-heavy, temporal-split evaluation and its
"tuning-free, laptop-runnable" framing sharpen the deployment-realism thread.
Directly relevant contrast for the **TFM vs trained-DL** debate: on medium/large
temporal data TabPack often matches or beats TabPFN-3 / TabICLv2 (which also
sometimes OOM), a useful counterweight to ICL-centric lectures. Also a clean case
study for the "ensembles as the real SOTA in tabular ML" point (TabArena's
Tuned+Ensembled regime).

## Caveats / limitations

- **Relies on the validation set** for both ensemble construction and early
  stopping; authors flag overfitting risk when val sets are small or shifted —
  which is exactly the enterprise/temporal regime, and plausibly why it lags on
  small IID TabArena datasets.
- **Not a foundation model / no cross-table transfer:** it's a better *trained*
  baseline, orthogonal to the relational/ICL thesis; contributes as a baseline,
  not as a bridge.
- **Higher training memory** than single models (11.3GB at 128 base models); needs
  reducing pack size on constrained GPUs.
- Gains over tuned TabM are modest in accuracy (rank 1.6 vs comparable) — the win
  is **compute/effort**, not a new accuracy frontier; the large-regression edge is
  from few datasets and partly un-tuned baselines.
- Engineering cost: new layers/optimizers must be reimplemented in "packed" form.

## Take-home

TabPack turns "tune an MLP ensemble" into "run it once": a single tuning-free run
matches heavily tuned tabular baselines and even runs on a laptop, making it a new
default trained-DL baseline — though it buys convenience/compute, not a new
accuracy ceiling, and leans on a trustworthy validation set.
