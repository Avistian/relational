Q3 Checkpoint: Write a Benchmark Report

Lesson 030 · Phase 1 · Q3 · ~50 min · capstone · primary reading: Grinsztajn et al. 2022, full

Q2 taught you to build the incumbent — a tuned, leak-free tree baseline. Q3 taught you to judge it: honest splits, leakage discipline, significance, the benchmark protocol and the three inductive biases (rotation, junk features) behind tree dominance, the honest neural baseline, and where AutoML stops. This checkpoint is not a new concept: it is the moment you prove you can assemble all of it into one defensible, reproducible benchmark report — the artifact a skeptic reads before believing any "model A beats model B" claim. Every result the thesis makes from here — that a relational model beats the flat-table incumbent — will be reported in exactly this form.

This checkpoint's single skill: assemble the whole Q3 toolkit into a one-page benchmark report — a curated dataset under an honest split, a leakage audit, the right contestants (a tuned GBDT, an honest neural baseline, and the AutoML bar) compared as a random-search budget curve rather than a single tuned number, a significance test that proves any gap is not noise, and an inductive-bias explanation of the ranking — ending in a verdict you would defend to a hostile reviewer, including the verdict "no significant winner."

One sentence. A benchmark report is trustworthy only when the comparison is fair (same data, split, metric, budget, preprocessing), the winner is proven (a correct significance test, not a bigger mean), and the result is explained (which inductive biases produced it) — and an honest report is willing to conclude that the gap is a statistical tie.

The benchmark-report contract

The Q2 fair-comparison contract fixed five things before touching a model (same data, split, metric, tuning budget, preprocessing scope) and reported the sixth honestly. Q3 adds three requirements that turn a fair comparison into a defensible report:

7 · The split must match deployment. A random (shuffled) split is only valid when the rows are genuinely i.i.d. If a time order or distribution drift exists, use a temporal split — train on the past, test on the future (L021). Random CV on time-ordered data leaked the deployment period into training and was optimistic (repro: random-CV 0.846 vs temporal 0.758). Default to temporal until you can argue i.i.d.
8 · Provenance is audited, not assumed. Fill a model info sheet (L022, Kapoor & Narayanan 2022): for every feature, is it knowable strictly before the label's timestamp? An illegitimate feature passes every split/pipeline check yet hands the model the answer, and a flexible model exploits it harder than a linear one — so a huge complex-vs-simple win is a leak hypothesis first (repro: gap +0.217 collapses to −0.009 when the leaked column is removed).
9 · The gap is proven, then explained. A reported gap is a random variable, so a "win" needs a significance test — and the obvious one lies. k-fold training sets overlap, so fold scores are positively correlated; a naive paired t-test (or Wilcoxon) treats them as independent and over-rejects. Use the corrected resampled t-test on one dataset (L023, Nadeau & Bengio 2003), or Friedman + Nemenyi across many. Then explain the ranking with the three inductive biases (L025–L027).
Why this is the checkpoint. The most common way a "relational model beats the baseline" claim collapses is an indefensible report — a random split on temporal data, an unaudited leaked feature, or a mean gap that a proper test dissolves. The whole thesis lives or dies by the quality of this one page. Learn to be the reviewer who would reject your own sloppy comparison.

The reference report: three families on credit_g

Here is the assembled benchmark, run under the full contract on real credit_g (1000 rows, 20 columns, 70% positive). Contestants: a GBDT (HistGradientBoosting, the Q2 incumbent), an honest neural baseline (a tuned MLP — the portable stand-in for the L028 tabular ResNet, same inductive biases), and the tiny AutoML from L029. credit_g has no timestamps, so a stratified split is defensible here — but the report states that rule explicitly rather than assuming it.

Section A — the budget curve, not one number

First, the L024 protocol: for each family, run a random search and plot the best-validation configuration's test score after each iteration. Held fixed: the split and preprocessing. Varied: the config drawn each step. Measured: the running best test AUC. This shows both the default (budget 1) and the tuned ceiling:

FamilyDefault (k=1)Ceiling (k=15)Read
GBDT0.8040.809strong, cheap default; tuning adds a hair
MLP (neural)0.8190.805tuning dipped — validation overfit

Read the honest wrinkle. The MLP's tuned ceiling (0.805) is below its default (0.819): a 15-config search, selected on a ~200-row validation slice, chased validation noise and picked a worse test model. That is not a bug — it is exactly Grinsztajn's "not too small" dataset criterion biting (L024): at this sample size the budget curve is noisy and can non-monotonically dip, which is itself a finding your report must disclose. Both families land in the same neighbourhood, ~0.80–0.82. A single tuned number would have hidden all of this.

Section B — the head-to-head, proven not asserted

A single split gives a fragile point estimate; run the families under one identical protocol and test the gap. Here is a paired 5×5 repeated stratified CV (25 folds, same folds for both families, default configs), then both significance tests from L023:

ModelROC-AUC (5×5 CV)Paired gapWhat it represents
GBDT (default)0.780—the Q2 incumbent
MLP (default)0.772−0.008the honest neural baseline
tiny AutoML≈ tie (L029)—ties a tuned GBDT — automation, not new accuracy

The mean paired gap is +0.0081 in the GBDT's favour (sd 0.0321 over 25 folds). Is that real? The two tests:

  • Naive paired t-test: t = 1.27, p = 0.22 — not significant.
  • Corrected resampled t-test (inflate the variance by 1/n + 1/(k−1) for the overlapping folds): t = 0.47, p = 0.64 — not significant, and even less so, as it must be (the correction can only make a gap harder to call significant, never easier).

The verdict, stated plainly: on credit_g, the GBDT is nominally ahead by ~0.008 ROC-AUC, but the gap is not statistically significant — a tie within noise. A report that wrote "GBDT wins" from the +0.008 mean would be making exactly the L023 mistake. Reporting the tie honestly is the deliverable.

Section C — explain the (small) lean

Even a tie needs an explanation, because the small edge that does exist leans the way the three biases predict. credit_g is small, single-table, heterogeneous, with several weak/near-junk columns — the regime where trees have a structural advantage:

  • Smoothness (L025): the target is irregular; the MLP's spectral bias over-smooths it, the piecewise-constant tree follows the jags.
  • Rotation (L026): the columns (credit amount, duration, age) are individually meaningful; the tree attends to each axis, the rotation-invariant MLP does not.
  • Uninformative features (L027): the tree gates junk columns out by gain; the MLP wires all of them into its first layer.

The three biases turn "GBDT edges it" from a mystery into a mechanism — and predict that the edge would grow on a larger, junkier single table and shrink on a clean, smooth one.

Read the honest half: a tie is a valid verdict

Beginners feel a benchmark must crown a winner. A mature report does the opposite: it reports the effect size and whether a correct test can distinguish it from noise, and here it cannot. That restraint is what makes the eventual relational claim credible — if this program will one day say "RDL beats the incumbent," a reviewer has to trust that the same authors would have said "no significant difference" when that was the truth. This page is the proof of that discipline. AutoML fits the same picture: L029 showed it only ties a tuned GBDT, so the single-table ceiling is not moved by cleverer search — the remaining upside has to come from a better representation.
