Manual Feature Engineering vs AutoML

Lesson 029 · Phase 1 · Q3 · ~50 min · core paper: Feurer et al. 2015, Auto-sklearn (skim)

The last five lessons taught us how to build strong contestants by hand: a tuned GBDT (L014–L017), a leak-free stacked ensemble (L018), and an honest neural baseline (L028). Each demanded human choices: which model, which hyperparameters, which features. Today's question is the natural one a skeptic asks: can a machine just do all of that for you? That is the promise of AutoML — and the canonical system is Auto-sklearn (Feurer et al. 2015). We will see exactly what it automates, what it does not, and run the fair fight that matters for the thesis: AutoML vs a hand-tuned XGBoost.

Its single skill: understand what AutoML automates — the CASH problem (jointly choosing the algorithm and its hyperparameters) solved by Bayesian optimization, plus meta-learning warm-start and automated ensemble construction — and run a fair AutoML-vs-tuned-XGBoost comparison, knowing that on typical tabular data AutoML usually ties a competently tuned GBDT at far higher compute, buys automation rather than a guaranteed win, and still does not invent your domain features or escape the single-table paradigm.

One sentence. AutoML automates model and hyperparameter search (and some preprocessing) — searching a combined space, warm-starting from similar past datasets, and ensembling the models it tried — but it does not replace domain feature engineering, and on typical tabular data it ends up roughly tied with a well-tuned GBDT, so its real value is convenience and robustness, not a new source of accuracy.

First, the vocabulary (nothing here is assumed)

This lesson names several ideas from the AutoML literature. Each is defined here, in plain language, before we rely on it. Where an idea is a callback to an earlier lesson, the definition is restated inline — do not assume you must remember it.

AutoML (automated machine learning). A system that takes a raw dataset and a metric and automatically produces a fitted model — choosing the algorithm, its hyperparameters, and (often) the preprocessing, without a human tuning each by hand. The goal is to turn the artisanal pipeline of L005–L018 into a push-button.
Hyperparameter optimization (HPO). Searching for the settings a model does not learn from data — a tree's depth, a GBDT's learning rate, an MLP's width. We met it in L017: grid search tries a fixed lattice, and random search (Bergstra & Bengio 2012) beats it at equal budget when only a few knobs matter, because it tries ~n distinct values on the important axis instead of √n.
Algorithm selection. The other half of the choice: which model family to use at all — logistic regression? random forest? gradient boosting? Traditionally a human picks this up front and then tunes it. AutoML folds it into the search.
CASH — Combined Algorithm Selection and Hyperparameter optimization. The problem Auto-sklearn solves, named by Thornton et al. (2013, Auto-WEKA). Treat "which algorithm" as one more (categorical) hyperparameter sitting above each model's own knobs, and search the whole nested space at once: A*, λ* = argmin over (A ∈ algorithms, λ ∈ hyperparams of A) of ValidationLoss( A(λ) ) So a single search jointly decides both the model and how it is configured — the core idea of this lesson.
Bayesian optimization & the surrogate model (SMAC). The search strategy AutoML uses instead of blind grid/random. It fits a cheap surrogate model (Auto-sklearn uses a random forest, in a method called SMAC) that predicts the validation score of a configuration without running it, plus its uncertainty. It then picks the next configuration to try by trading off exploitation (looks good) against exploration (uncertain, could be good) — an acquisition function. The effect: it spends its budget where the payoff is likeliest, so it usually needs fewer trials than random search to reach a good configuration. (In our lab we use random search as an installable stand-in — the mechanism you must know is CASH + select-by-validation; SMAC is the smarter way to pick the next point.)
Meta-learning, meta-features, warm-start. "Learning across datasets." Auto-sklearn describes each dataset by a vector of meta-features — cheap summary statistics (number of rows, number of features, class-imbalance ratio, skew, etc.). Offline, it records which configurations worked well on ~140 reference datasets. Faced with a new dataset, it finds the nearest reference datasets by meta-features and warm-starts the Bayesian search from their winning configurations — so it does not begin from scratch.
Ensemble selection (Caruana et al. 2004). A search leaves behind a whole pool of trained models, not just the winner. Ensemble selection builds a weighted average greedily: start empty, then repeatedly add the pool model that most improves the ensemble's validation score, with replacement (a strong model can be added several times, giving it more weight). It is the leak-free cousin of the stacking from L018 — no meta-learner to overfit, just a validated weighted blend.
Feature engineering (manual). Turning raw columns into informative ones using domain knowledge — ratios, date parts, aggregates over related rows (L009). This is the step AutoML systems like Auto-sklearn largely do not do: they search models and generic preprocessing, not domain-specific feature construction. Keep this distinction sharp — it is where the lesson lands.

What AutoML automates (1): the CASH search

Start with the heart of Auto-sklearn: it searches a combined space. In the paper this space is large — roughly 15 classifiers, about a dozen feature-preprocessing methods, a few data-preprocessing methods, and ~110 hyperparameters in total, all wired into one structured configuration space. A single Bayesian search then hunts for the (algorithm, hyperparameters) pair with the best validation score — solving the CASH problem defined above.

To see it concretely, here is a real 40-iteration search we ran over a small CASH space — four algorithms (logistic regression, random forest, extra-trees, HistGB) and their hyperparameters — on credit_g. Held fixed: the preprocessing and the train/validation split. Varied: the algorithm and its hyperparameters, drawn each iteration. Measured: the validation ROC-AUC of each configuration, and the best found so far. Filter by algorithm to see how the budget is spread.

Read the picture. The search does not commit to one model family — it evaluates configurations in every algorithm lane (that is the "combined" in CASH). The black step line is the best-validation-AUC-so-far, and it climbs with budget exactly like the L024 budget curve: 0.796 → 0.817, reaching its peak at iteration 15 with a HistGB configuration. The single rule doing all the work is the one from L004/L017: keep the configuration with the best validation score. AutoML's contribution is to run that search over models and knobs together, and (with SMAC) to choose the next point cleverly.

What AutoML automates (2): meta-learning warm-start

A cold Bayesian search wastes its first several trials orienting itself. Auto-sklearn's first extension over Auto-WEKA removes that waste: using the meta-features defined above, it identifies which past datasets most resemble the new one and warm-starts the search from the configurations that won there. Think of it as starting the climb near a previously discovered summit instead of at sea level. It does not change what is being searched (still CASH); it changes where the search begins, which matters most under a tight time budget. (Our small demo has no meta-learning store, so its curve starts cold — the real system's early iterations would be better.)

What AutoML automates (3): ensemble selection

A search trains dozens of models and then, naively, throws all but the best away. Auto-sklearn's second extension keeps them: it runs greedy ensemble selection (Caruana 2004) over the pool. Here is that procedure on the same credit_g search — the single best-validation model versus the greedy weighted blend. Toggle between them.

The single best configuration (a HistGB) scores 0.824 on the held-out test set. The greedy ensemble — built by repeatedly adding whichever pool model most improves the validation AUC, with replacement — ends up a blend of three different algorithms (LogReg, HistGB, ExtraTrees) and scores 0.831, a +0.007 gain for free: those models were already trained during the search. The mechanism is the same as L018 stacking — diversity plus a leak-free selector — but simpler (a weighted average chosen on validation data, no meta-learner to overfit). Note the honest caveat from L018: on a 1000-row single table the ensemble gain is real but small; ensembling is not magic.

Predict first

We are about to pit three things against each other on credit_g, scored by ROC-AUC over 5 seeds: an untuned default XGBoost, a hand-tuned XGBoost (random search over its hyperparameters, L017 discipline), and the tiny AutoML (CASH search + ensemble selection) built from the mechanisms above. The honest bake-off: AutoML ties a tuned XGBoost

Here is the result. Bars are mean test ROC-AUC over 5 seeds; whiskers are ±1 standard deviation.

ModelROC-AUC (mean ± sd)What it represents
default XGBoost0.775 ± 0.025no search at all — out-of-the-box settings
tuned XGBoost0.806 ± 0.016manual: random search over XGB's own hyperparameters
tiny AutoML0.803 ± 0.020CASH search across 4 algorithms + greedy ensemble

Two readings, and hold both:

  • The big jump is tuning, not AutoML. Going from the untuned default (0.775) to a tuned XGBoost (0.806) buys +0.031 — most of the available gain. The lesson of L017 stands: searching at all is what matters. Reporting an untuned model is the same mistake as the weak neural baseline in L028.
  • AutoML ties the hand-tuned XGBoost. The tiny AutoML (0.803) sits inside the tuned XGB's error band (−0.002; the whiskers overlap). It searched four algorithms and ensembled them, at several times the compute — and landed in a statistical tie (echo L023: a gap this small, with these error bars, is not a real difference). It did not lose — but it did not unlock accuracy a competent human with one strong model family could not.

This is the honest, and slightly deflationary, finding. On typical tabular data a well-tuned GBDT is already close to the ceiling; AutoML's value is that it reaches that ceiling with far less human effort and more robustly across datasets — you do not have to know that a GBDT is the right family, or tune it well, or remember to ensemble. That convenience is genuinely valuable (it is why Auto-sklearn won the ChaLearn AutoML challenge). It is just not a new well of predictive power.

Explain it back

This is the load-bearing idea — put it in your own words before moving on.

Thesis bridge. AutoML is the strongest possible version of "just automate the single-table pipeline," so where it stops is exactly where the relational bet begins. Three consequences. (1) It sharpens the bar, on one axis only. An honest single-table baseline can now be "whatever AutoML finds," which forecloses the cheap objection that an RDL win merely beat a lazily-tuned model — but AutoML searches models and knobs, never the feature representation, so it cannot rescue signal the flattening step already destroyed (C1). (2) It automates the wrong half. Auto-sklearn explicitly does not do domain feature engineering; on relational data the highest-value features are cross-table aggregates (L009, DFS/Featuretools), and constructing those by hand is precisely the lossy, leakage-prone labor (L002) that RDL proposes to learn end-to-end (C2). AutoML tunes the model on top of a flattened table; RDL changes the table. (3) The tie is the tell. That a four-algorithm AutoML only ties a tuned GBDT confirms L024–L028: within the single-table world the returns to cleverer search are nearly exhausted, so the remaining upside must come from a better representation — which is the thesis (C4).

Where this is subtle (know this too)

  • AutoML ≠ automated feature engineering. Auto-sklearn searches algorithms, hyperparameters, and generic preprocessing (scaling, PCA, feature selection). It does not invent domain features. Systems that do generate features — Featuretools/DFS (L009), later AutoFeat — are a different tool; they are the manual bridge RDL later learns.
  • Random search is our installable stand-in for SMAC. Auto-sklearn's real search is Bayesian (SMAC), which is more sample-efficient per trial. We used random search in the demo/lab because it installs anywhere and Bergstra & Bengio (L017) show it is a strong baseline. The mechanism to remember is CASH + select-by-validation + ensemble, not the exact optimizer.
  • Compute is not free. The AutoML result cost several times the wall-clock of the tuned XGBoost for a tie. On a fixed budget, spending it all tuning one strong family can beat spreading it across many — an honest cost accounting is part of the comparison (L024's budget-curve mindset).
  • Same evaluation discipline applies. AutoML can leak just as easily as a human: it must select on a proper validation split, respect temporal/grouped structure (L021), and be scored on a truly held-out test set. "The machine chose it" is not a defence against leakage.
  • "No free lunch" for search, too. AutoML's edge is largest when you don't already know the right model — on heterogeneous problems, or for non-experts. When you know a tuned GBDT is the answer (as here), the automation mostly saves labor.
