Inductive Bias: Uninformative Features

Lesson 027 · Phase 1 · Q3 · ~45 min · core paper: Grinsztajn, Oyallon & Varoquaux 2022, §5.3

This is the third and final mechanism lesson of the Grinsztajn arc. Lesson 024 built the fair benchmark; Lesson 025 dissected the first inductive bias (smoothness) and Lesson 026 the second (rotation). Today we take the third: tree-based models are robust to uninformative features, and neural networks are not. It is the most practical of the three — real tables are full of columns that carry little or no signal — and it is tied directly to the rotation bias by a theorem we met last lesson.

Its single skill: explain why adding uninformative ("junk") features degrades an MLP far faster than a tree, name the mechanism that makes a tree robust (greedy, gain-gated split selection = implicit feature selection), connect it to Ng's rotation-invariance bound from L026, and recognize the two experiments that prove it — adding junk features widens the tree–MLP gap, and removing them shrinks it.

One sentence. A decision tree only ever spends its capacity on features that reduce impurity, so a pure-noise column is quietly ignored; a neural network feeds every column into its first layer and — because it is rotationally invariant — cannot cheaply tell a useless direction from a useful one, so each junk feature costs it real accuracy.

First, the vocabulary (nothing here is assumed)

This lesson leans on a few terms. Rather than send you back to earlier lessons, here is each one in plain language, defined before we use it.

Uninformative (junk) feature. A column that carries no information about the label — statistically independent of the target. In our experiments it is a pure-noise column: values drawn from a Gaussian that has nothing to do with y. Real tables are full of near-junk columns (an internal ID, a mostly-constant flag, a measurement unrelated to the outcome). The question of this lesson is: what does a model do when you hand it a pile of such columns alongside the real ones?
Impurity & Gini (recall from L011). A node's impurity measures how mixed its labels are. The Gini impurity of a binary node with a fraction p of positives is 1 − p² − (1−p)² — it is 0 when the node is pure (all one class) and 0.5 when the classes are evenly split. A tree wants to cut the data into purer and purer regions.
Split gain. When a tree considers splitting a node on some feature at some threshold, the gain is how much impurity the split removes: gain = parent_impurity − weighted average of the two children's impurities. A split that separates the classes has a large gain; a split on noise leaves the children as mixed as the parent, so its gain is ≈ 0. A tree is greedy: at every node it picks the feature and threshold with the highest gain. This is the gate at the centre of today's lesson.
Implicit feature selection. Because the tree only ever splits on the highest-gain feature, a junk column — whose best split has ≈ 0 gain — is simply never chosen (at least near the top of the tree). The tree selects the useful features as a side-effect of how it grows, without any separate feature-selection step. Neural networks have no such built-in step.
Rotational invariance & Ng's bound (recall from L026). A learner is rotationally invariant if applying the same rotation (an orthogonal matrix Q, one that preserves all distances and angles) to the train and test features leaves its accuracy unchanged. An MLP is invariant because its first layer is a full linear map: W·(Qx) = (WQ)·x, so it can absorb the rotation into its weights. Ng (2004) proved the consequence we need today: a rotationally-invariant learner needs a number of training examples that grows at least linearly in the number of uninformative features. Keep that sentence handy — it is the bridge from L026 to this lesson.

The claim, and how the paper isolates it

Grinsztajn's Finding 2 (their §5.3) is stated as: "MLP-like architectures are not robust to uninformative features." Proving a claim like this cleanly requires an ablation — a controlled experiment where you change exactly one thing and hold everything else fixed, so any change in the result can only be attributed to the thing you changed. The paper runs two complementary ablations:

  1. Add uninformative features. Take a dataset, append extra pure-noise columns, and refit. If MLPs are less robust, adding junk should hurt them more than the trees, so the gap should widen.
  2. Remove uninformative features. Rank the existing features by importance and drop the least useful ones. If MLPs were disproportionately hurt by junk, removing it should help them more than the trees, so the gap should shrink.

We reproduce both on a synthetic task so we control exactly how much junk is present. The setup — read this carefully, because the interpretation depends on it:

  • Held fixed: 5 informative features and the label. The label is a smooth, mildly-nonlinear function of those 5 features (y = 1 when 1.3x₁ + x₂ + 0.8x₃ + 0.6·x₄x₅ is above its median).
  • Varied: the number of extra pure-noise columns k (0 → 100), each an independent Gaussian with zero relationship to y.
  • Measured: test accuracy of a decision tree, a gradient-boosted tree (GBT), and an MLP, averaged over 5 seeds.
  • Why this isolates this bias: we deliberately chose a smooth target so that on clean data (k = 0) the MLP wins. That rules out the smoothness bias (L025) and the rotation bias (L026) as the explanation — if the MLP starts ahead and then loses the lead, only the junk features can be responsible.

Why is a tree robust and an MLP not? The gate.

The mechanism is the greedy, gain-gated split selection we defined above. At each node a tree scores every candidate split by its gain and keeps only the best. On the full data, a pure-noise column's best possible split barely reduces impurity, so it loses the competition to a real feature and is never chosen near the top. The tree's structure is therefore anchored by the informative features; junk columns are gated out for free.

An MLP has no such gate. Every input feature is wired into the first linear layer, and — because the MLP is rotationally invariant — it has no built-in notion that "this axis is a real feature and that one is noise" (a rotation can turn one into a mix of the others). It must learn to down-weight the junk directions from the data, and Ng's bound says that costs samples: at least one more chunk of training data per useless feature. With a finite training set, the leftover weight on junk directions is just extra variance — the network partly fits the noise, and test accuracy drops.

We can see the gate directly. On a dataset with 5 informative + 20 junk features, we measured, for each feature, (left) the tree's root-split gain — the impurity it could remove with its single best threshold on the full data — and (right) the trained MLP's first-layer weight magnitude — how much of its input weight it spends on that feature. Each panel is normalized so the informative bar is 1.0; the junk bar shows the fraction the model still spends on noise.

The contrast is the whole point. The tree's best split on a junk column removes about 1% as much impurity as its best split on an informative column — a ~118× gate — so the greedy rule never picks it. The MLP, with no gate, still puts about 56% as much first-layer weight on junk as on signal — barely a 1.8× difference. The tree selects; the MLP dilutes.

Honest subtlety (worth knowing). Trees are not perfectly blind to noise. Deep in a tree, after the data has been split many times, a small subset can show a spurious, high-gain split on a junk column purely by chance — which is exactly why impurity-based (MDI) feature importance is known to over-credit noise features. That is why we measured the root-split gain on the full data (where the gate is real) rather than MDI. The accuracy numbers confirm the gate is what matters: those deep spurious splits are low-gain and get averaged away by boosting/bagging, so the trees' accuracy barely moves. The gate is a strong tendency, not a theorem — but it is more than enough to make trees robust.

The theory bridge: Ng (2004), spelled out

Last lesson we met Ng's theorem as a one-liner. It is worth unpacking, because it explains why the MLP's cost grows the way it does — and it links the rotation bias (L026) to the junk-feature bias (L027) as two faces of the same coin.

Ng studied learners that are rotationally invariant (MLPs, logistic regression with L2 regularization, kernel methods with a rotation-invariant kernel). His result: for such a learner, the number of training examples needed to reach a fixed accuracy grows at least linearly in the number of irrelevant (uninformative) features. The intuition is exactly the gate story, told from the sample-complexity side. To ignore a junk direction, an invariant learner must first figure out which direction is junk — but rotation invariance means it treats all directions symmetrically at the start, so it has to estimate the orientation of the useful subspace from data. Each extra junk dimension adds one more direction it could mistakenly lean on, and pinning down "don't use this one" takes more examples. By contrast, a method that is not rotation-invariant — like L1-regularized logistic regression, or a tree — can exploit the axis structure to reject junk features with far fewer samples (Ng's paper is literally titled "…L1 vs L2 regularization, and rotational invariance").

The two biases are one idea. Rotation invariance (L026) says an MLP cannot tell a meaningful axis from an arbitrary mixture. Uninformative-feature fragility (L027) is the price of that: if you cannot tell which axis is meaningful, you cannot cheaply tell which axis is meaningless either. A model that respects the natural basis of a table gets both robustness properties for free.

Explain it back

This is the load-bearing idea — put it in your own words before moving on.

Thesis bridge. This bias, like rotation, cuts both ways for the relational bet. Against a naïve version: a relational model that ends by pooling a graph into a flat vector and feeding an MLP head inherits the MLP's fragility to junk — and a database has lots of near-junk columns (surrogate keys, audit timestamps, denormalized duplicates). "Add a GNN, then an MLP" does not automatically get you the tree's implicit feature selection; you must design for it. For the bet: the deeper reading is that a good model should select structure, not drown in it — and relational learning is precisely a bet on selection at the level of tables and relationships, not just columns. A message-passing model that learns which foreign-key paths carry signal is doing implicit feature selection over the schema, the same instinct that makes a tree ignore a noise column, scaled up to the whole database. On flat tables the field's fix is telling: numeric-feature embeddings and attention (SAINT, FT-Transformer) let a neural model learn to attend to the informative features — a learned gate that recovers much of the robustness a tree gets for free, and a preview of Year 2.

Where this is subtle (know this too)

  • It is about relative robustness, not immunity. Enough junk hurts a tree too (our GBT lost 0.032). The claim is that it hurts the MLP more — here more than twice as much — which is enough to flip the ranking.
  • The effect compounds with data size. Ng's bound is about sample complexity, so with a very large training set an MLP can learn to ignore junk — it just needs the extra examples. This is a small/medium-data phenomenon, like the rest of the tree advantage.
  • Removing junk is a real, cheap win for neural nets. The remove-junk direction of the experiment is not just theory: feature selection (or letting the model learn to attend) helps a neural network disproportionately. If you must use an MLP on a wide table, prune uninformative columns first.
  • MDI importance over-credits noise. Do not use impurity-based feature importance to argue "the tree ignores junk" — it does the opposite near the leaves. Use permutation importance, or (as we did) root-split gain, when you want to see the gate.
  • This is one of three biases. Uninformative features (L027) compound with smoothness (L025) and rotation (L026); on a typical table all three push the same way at once. Do not over-attribute the whole tree advantage to any single one — Grinsztajn's point is that typical tabular data has all three properties together.
