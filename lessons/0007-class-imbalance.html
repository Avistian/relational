Class Imbalance

Lesson 007 · Phase 1 · ~40 min

Every leakage lesson so far — 002, 005, 006 — protected the honesty of your score. This lesson protects something the score itself can hide. When one class is rare — churn, fraud, conversion, a positive diagnosis — a model can be 95% accurate and completely useless, and a careless fix can leak the very signal you're measuring.

This lesson's single skill: on an imbalanced problem, stop trusting accuracy, judge with imbalance-aware metrics, and apply the handling menu (class weights, threshold tuning, resampling) without leaking — which means any resampling lives inside an imblearn Pipeline, exactly as your transforms did in Lesson 005.

The accuracy paradox

Take a dataset that is 5.5% positive. A model that ignores the input and always predicts "negative" scores like this (verified, sklearn 1.9):

ModelAccuracyRecall (positives caught)PR-AUC
DummyClassifier("most_frequent")0.9450.0000.055

94.5% accuracy, and it catches zero positives. Accuracy just measures the majority class back to you. This is the accuracy paradox: on imbalanced data, the headline metric rewards a model for doing nothing (He & Garcia 2009).

Metrics that survive imbalance

Replace accuracy with metrics that look at the rare class directly:

MetricWhat it answersUse when

F1harmonic mean of precision & recallyou need one balanced number

ROC flatters, PR tells the truth. Verified on the same model: ROC-AUC 0.914 vs PR-AUC 0.747. ROC-AUC uses the true-negative-rich denominator, so it looks rosy when negatives dominate. For rare positives, lead with PR-AUC. (We go deep on metrics next lesson — here just stop using accuracy.)

The handling menu

Three families, roughly in order of "reach for this first":

  1. Cost-sensitive learning (class weights). Tell the model a positive error costs more. In sklearn, class_weight="balanced"; in XGBoost, scale_pos_weight. No data is added or removed, so there is nothing to leak — this is the safe default. Verified effect (logistic regression): recall 0.55 → 0.83.
  2. Threshold tuning. Keep the model; move the decision threshold to match your cost trade-off. Choose it on validation data, never on test. Verified: F1 rose from 0.711 at the default 0.5 to 0.725 at threshold 0.3.
  3. Resampling. Rebalance the training set — undersample the majority, or synthesize minority points with SMOTE (Chawla et al. 2002). Powerful, but it rewrites your data — and that is where the trap is.
A caution the metric won't show. Class weights changed recall a lot (0.55 → 0.83) but barely moved PR-AUC (0.756 → 0.734). Reweighting and resampling mostly shift the operating point — they trade precision for recall — rather than making the model rank better. Don't expect them to manufacture signal that isn't there; expect them to move where you sit on the precision/recall curve.

The resampling leak (this is the exam question)

Here is the mistake that quietly inflates a huge fraction of "imbalanced-data" results. SMOTE creates synthetic minority points by interpolating between real ones. If you SMOTE the whole dataset and then cross-validate, synthetic points in your test folds are interpolations of points in your training folds — the model is graded on near-copies of what it trained on. Worse, it's also tested on an artificially balanced set it will never see in production (imbalanced-learn, Common pitfalls). 

# WRONG — SMOTE sees every row, then we cross-validate on it
X_res, y_res = SMOTE().fit_resample(X, y)
wrong = cross_val_score(clf, X_res, y_res, cv=cv, scoring="f1").mean()

# RIGHT — SMOTE lives in an imblearn Pipeline; it touches train folds only
from imblearn.pipeline import make_pipeline
from imblearn.over_sampling import SMOTE
pipe  = make_pipeline(SMOTE(random_state=0), clf)
right = cross_val_score(pipe, X, y, cv=cv, scoring="f1").mean()

Verified (imbalanced-learn 0.14): the leak invents 0.41 of F1 out of interpolation — the same shape of lie as Lesson 005's feature-selection demo (0.78 vs 0.44 on pure noise). The rule generalizes: anything that uses the data to transform the data belongs inside the cross-validation fold.

One wrinkle Lesson 005 didn't have: SMOTE changes y, not just X. A plain sklearn.pipeline.Pipeline can't host a step that resamples rows. That's why imblearn ships its own Pipeline — same interface, but it routes resamplers correctly and, crucially, skips them at predict time so the held-out fold keeps its natural imbalance.

Bridge to the thesis

Imbalance is not an edge case for relational deep learning — it is the default. The headline RelBench tasks (churn, fraud, conversion, link prediction) all have rare positives. Three consequences for a fair baseline:

  • Report PR-AUC / AUROC, not accuracy. A churn model at 95% accuracy that predicts "nobody churns" is the accuracy paradox in production. RelBench leaderboards are scored on ranking metrics for exactly this reason.
  • Reweight, don't resample, across a temporal split. scale_pos_weight / class_weight is leak-free; SMOTE before a time-based split is the resampling leak plus a time leak. Your tuned GBDT baseline should use weights.
  • SMOTE barely makes sense on an entity graph. What is the interpolation of two customers, with their orders and reviews? Synthesizing rows fractures relational structure. RDL instead keeps every real entity and lets the loss carry the class weights — a cleaner story the thesis can lean on.

Four mistakes to catch

  1. Reporting accuracy on imbalanced data. Rewards predicting the majority. Fix: PR-AUC, recall, balanced accuracy.
  2. SMOTE before the split. Synthetic test points leak from train. Fix: SMOTE inside an imblearn Pipeline.
  3. Tuning the threshold on the test set. Leaks the answer. Fix: pick the threshold on validation.
  4. Expecting reweighting to add signal. It shifts the operating point, not the ranking. Fix: judge with PR-AUC, set the cost deliberately.
