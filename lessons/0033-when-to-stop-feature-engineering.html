When to Stop Feature Engineering

Lesson 033 · Phase 1 · Q4 · ~40 min · primary reading: Domingos 2012, “A Few Useful Things to Know about Machine Learning”

You now know how to build features (L009), encode categoricals four ways (L031), and even let a Transformer contextualise them (L032). Every one of those is a way to spend human effort shaping the input a model sees. This lesson is about the question nobody teaches but every practitioner faces: when do you stop? You have a fixed number of hours; some go to crafting features, some go to tuning the model. Pedro Domingos' famous 2012 essay gives the two facts that decide the split — “feature engineering is the key” and, in the same breath, “more data beats a cleverer algorithm” and “overfitting has many faces” — and this lesson turns them into a rule you can act on.

Its single skill: allocate a fixed modeling budget between feature engineering and model tuning, and justify the split with the feature-engineering trade-off — most of the early gains live in the features, but their returns diminish and each added feature raises the overfitting/variance tax, so you stop when the marginal held-out gain sinks into the cross-validation noise band rather than when you run out of ideas.

One sentence. Feature engineering is the highest-leverage thing you can do early, but on a strong model and a small flat table its returns diminish fast and eventually go negative (overfitting) — so the disciplined move is to measure the curve of held-out score vs. effort, watch it against the CV noise band, and stop where the marginal gain is no longer distinguishable from noise.

First, the vocabulary (nothing here is assumed)

This lesson leans on a few terms about effort, returns, and risk. Each is defined here in plain language before we rely on it — assume you are meeting each one fresh.

Feature engineering (FE). Hand-crafting new input columns from the raw ones — ratios, products, logs, aggregates, flags — so the structure a model needs becomes easier for it to represent (L009). A hand feature is one such crafted column. FE is human work: it encodes your domain knowledge into the design matrix.
Marginal return (of a feature). The change in a model's honest held-out score caused by adding one more feature, holding everything else fixed. If baseline score is \(s_0\) and adding a feature gives \(s_1\), the marginal return is \(s_1 - s_0\). “Diminishing returns” means each successive feature's marginal return tends to shrink.
Diminishing returns. The pattern where the first few features buy most of the achievable gain and later ones buy progressively less — the curve of score-vs-effort flattens. In the extreme it turns negative: adding features starts to lower the held-out score.
Overfitting & the bias–variance trade-off. A model overfits when it fits noise particular to the training sample and so generalises worse. Adding features raises variance (more ways to fit noise) while lowering bias (more ways to fit signal); past a point the variance cost outweighs the bias gain and held-out score falls. Domingos calls this “overfitting has many faces” — it is not one bug but a family of them.
Cross-validation noise band (±1σ). A CV score is a mean over folds, and it has a standard deviation \(\sigma\) across those folds (L003). The “noise band” is the interval mean \(\pm\,\sigma\). A change smaller than \(\sigma\) is within noise — you cannot tell it from luck of the split (the discipline of L023). Real improvements clear the band; noise does not.
Modeling budget & opportunity cost. The fixed time/compute you have for a project. Every hour spent engineering a feature is an hour not spent tuning the model, getting more data, or auditing leakage — that forgone alternative is the opportunity cost. Allocating the budget well means putting each hour where its marginal return is highest.
“More data beats a cleverer algorithm.” Domingos' observation that, past a competent model, adding training data usually raises held-out score more per unit effort than adding model cleverness or more hand features — because more data reduces variance directly. A reason the FE budget is not the only, or even the best, place to spend.

Domingos' three “useful things” that decide the budget

Domingos (2012) distils a dozen folk-truths of applied machine learning. Three of them, read together, settle how to split a modeling budget. We take each at its word and then test it on real data below.

Domingos' lessonWhat it claimsConsequence for your budget
“Feature engineering is the key” Across projects, the features used matter more than the algorithm; FE is where most of the time and most of the early gains go. Spend the first hours on features, not on exotic models.
“More data beats a cleverer algorithm” Once the model is competent, extra data usually beats extra cleverness or extra features per unit effort. Before hand-crafting a 10th feature, ask whether more rows would pay more.
“Overfitting has many faces” More features / capacity raise variance; held-out score eventually falls, and it can fall in subtle, hard-to-spot ways. There is a stopping point — past it, more FE is negative-return.

Notice the built-in tension: the first lesson pushes you toward feature engineering; the second and third pull you back. The skill is holding both — FE is the highest-leverage thing early, and a trap late. The only way to know where “early” becomes “late” is to measure the returns.

Why feature engineering matters at all (the case for spending early)

Before we talk about stopping, remember why you start. From L009: a model can only combine features in the ways its hypothesis space allows. A linear model can only add weighted columns; it cannot, on its own, synthesise a ratio of differences \((a-b)/(c-d)\). Engineer that ratio and the same linear model goes from a shapeless cloud to a near-perfect line. Toggle the view below (the L009 result, re-shown).

Verified earlier (L009, target \(y=(a-b)/(c-d)\), 5-fold R²): a Ridge regressor scores R² 0.644 on the raw columns and 0.998 once the ratio is engineered in. This is “feature engineering is the key” in its purest form — but note it was most dramatic for a weak (linear) model. A gradient-boosted tree, which already approximates interactions through its splits, rose only 0.968 → 0.980. Hold that asymmetry; it is exactly what the next experiment exploits.

The experiment: watch the returns diminish (and go negative)

Let us be explicit about the setup, because the design is what makes the result trustworthy (L022 discipline):

  • Held fixed: the model (HistGradientBoosting, default hyperparameters), the data (credit_g, 1000 rows), the 5-fold stratified CV, and the seed. Only the feature set changes, so any movement is attributable to features alone.
  • Varied: the number of hand features \(k = 0,1,\dots,10\), added in a fixed, sensible order an analyst would try first (monthly payment = amount/duration, amount/age, installment×amount, amount×duration, …). The order is chosen in advance, not greedily on the label — so the curve is an honest “as effort accrues” trace, not a cherry-picked optimum.
  • Measured: mean 5-fold CV ROC-AUC and its standard deviation \(\sigma\) (the noise band) at each \(k\).

Drag the slider to spend more feature-effort. Watch the green (GBDT) curve and its shaded ±1σ band. The dashed orange line marks the honest stopping point.

Read it honestly, row by row. The GBDT baseline (\(k=0\)) is 0.7865. The best it ever reaches is 0.7911 at 3 features — a gain of +0.0046, which is well inside the ±0.032 noise band, i.e. not distinguishable from luck of the split (L023). Push past three and the curve declines: by 8 features it is 0.7659, below the no-feature baseline — the overfitting tax, live. The linear model (toggle it on) tells the complementary story: it starts higher (this dataset is fairly linear) and barely moves, drifting +0.006 total, also inside its band. On this table, against a competent model, single-table feature engineering buys nothing you can prove — and past a handful of features it starts to cost you.

“Within noise” is the whole point. The temptation is to see 0.7911 > 0.7865 and declare the three features a win. But a +0.0046 mean inside a ±0.032 band is exactly the L023 / L030 mistake: celebrating noise. The correct report is “no measurable single-table FE gain on credit_g,” and the correct action is to stop — the marginal return has sunk below what the data can resolve.

The honest verdict: FE's value depends on the model and the data

SituationReturn on hand featuresWhy
Weak model (linear) · feature it can't synthesiselarge (L009: 0.64 → 1.00)you supply the hypothesis the model lacks

So “feature engineering is the key” is not wrong — it is conditional. It is the key when the model is weak relative to the structure, or when the structure isn't yet in the columns at all. On a strong model already fed the informative columns, single-table FE hits diminishing returns almost immediately, and the honest budget rule follows.

The budget rule you can act on

Turn the curve into a decision procedure. Given a fixed budget, spend it greedily on whichever action has the highest marginal held-out return, and stop each activity when its marginal return drops below what you can measure:

  1. Front-load features. Spend the first block engineering the handful of features your domain knowledge says matter — the steep part of the curve. Measure CV score with its σ after each.
  2. Stop when Δ < σ. When the marginal gain of the next feature is smaller than the CV standard deviation (it has entered the noise band), stop adding features — you can no longer prove it helps, and you are courting the overfitting tax.
  3. Reallocate. Move the remaining budget to the action with the higher marginal return: usually more data (Domingos' second lesson), then model tuning (L017), then an honest significance check (L023) before you believe any of it.
  4. Never “finish” by exhausting ideas. The wrong stopping rule is “I ran out of features to try.” The right one is “the held-out curve flattened into the noise.” They are rarely the same point.

Explain it back

This is the load-bearing idea — put it in your own words before moving on.

Thesis bridge. This lesson is quietly one of the most important for the thesis, because it names the ceiling. (1) Manual FE is bounded, and cheaply so. On a strong model and a single flat table, hand features hit diminishing (then negative) returns almost immediately — the value you can add by reshaping one table is nearly exhausted, exactly as L028–L032 showed for search, architecture, and representation. (2) The unclaimed value is structural, not effort-based. The features that would pay are aggregates across related tables — “customer's average transaction over the last 90 days,” “number of prior defaults” — which no amount of single-table cleverness can reach. Deep Feature Synthesis (the DFS idea from L009) automates exactly those relational aggregates by hand; relational deep learning aims to learn them, end-to-end, instead of engineering them. (3) The human-effort ratio is the real prize. Domingos' “feature engineering is the key” is implicitly a statement about human hours; the RDL bet (tested directly in Year 4's manual-FE-vs-RDL studies) is that a learned relational model matches or beats hand-built features at a fraction of the human effort. Where this lesson says “stop hand-engineering — the single-table returns are gone,” the thesis answers “because the returns moved across the join.”

Where this is subtle (know this too)

  • “Within noise” ≠ “useless.” A feature that doesn't move AUC can still be worth keeping for interpretability, robustness under drift, or because it makes a leak-safe pipeline simpler. The budget rule is about accuracy returns; other objectives have their own curves.
  • Model-dependence is the headline, not a footnote. The same feature can be worth +0.35 R² to a linear model and ≈0 to a GBDT (L009 vs this lesson). Always ask “diminishing returns for which model?” A tree synthesises interactions; a linear model cannot — so FE budgets differ by model family.
  • Fixed-order vs. greedy selection. We added features in a pre-chosen order to trace an honest effort curve. Choosing the order greedily on the label would look better but is a form of selection that must live inside the CV fold, or it leaks (L022) — a peaked greedy curve on the full data is optimistic.
  • More data is the usually-better lever. Before the 8th hand feature, Domingos says check whether more rows would pay more. Variance (the thing overfitting inflates) falls with sample size; on a 1000-row table, doubling the data often beats any single engineered column.
  • FE can also destroy signal or leak it. A careless aggregate can compute over the future (point-in-time leakage, L021/L022) or collapse a useful distinction. “Add features until it stops helping” silently assumes each feature is leak-safe; verify provenance, don't just count.
