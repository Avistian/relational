Inductive Bias: Rotation

Lesson 026 · Phase 1 · Q3 · ~35 min · core paper: Grinsztajn, Oyallon & Varoquaux 2022, §5.4

Lesson 024 built the benchmark; Lesson 025 opened the mechanism arc with the first of Grinsztajn's three inductive biases — smoothness. This lesson takes the second spatial bias, and the most surprising one: rotation. It is the bias that explains why the very structure of a table — the fact that each column means something on its own — favours trees, and it is the one that most directly points at the relational thesis.

Its single skill: explain rotational invariance in the sense of Ng (2004) — why a decision tree is not invariant while an MLP is, why that invariance is a liability on tabular data whose columns carry individual meaning, and recognize the experiment that proves it: randomly rotating the feature space leaves the MLP untouched but destroys the tree's edge, and can reverse the ranking entirely.

One sentence. A decision tree attends to each feature on its own axis, so it exploits the fact that a column like age or balance is individually meaningful; an MLP treats the features as an unlabelled vector and is unchanged if you rotate the axes — which sounds elegant but is exactly wrong for tabular data, where the original basis is the signal.

What "rotational invariance" means

Take your feature matrix X and multiply it by a random orthogonal matrix Q (a rotation: it preserves every distance and angle, just spins the axes). Apply the same Q to train and test. A learning procedure is rotationally invariant (Ng, 2004) if training on X and training on XQ give you the same test performance.

rotate: X → X·Q, where QᵀQ = I (train and test rotated by the same Q)

An MLP is (essentially) rotationally invariant: its first layer is a full linear map W·x, and W·(Qx) = (WQ)·x, so the network can absorb any rotation into its weights and reach an equivalent solution. A ResNet is invariant for the same reason. A decision tree is not: it splits on one feature at a time at an axis-aligned threshold, so its whole vocabulary is tied to the original coordinate system. Rotate the axes and "feature 3 > 0.5" no longer means anything. (An FT-Transformer is also not invariant — its feature tokenizer is a per-column operation — which will matter at the end.)

See it: why a tree can't survive a rotation (§5.4)

Here is an axis-aligned class region — the top-right quadrant (feature 1 > 0 and feature 2 > 0). The tree (left) carves it with two straight, axis-aligned splits and nails it. Press Rotated basis to spin the whole space: the region becomes a diagonal wedge, and the tree — still forced to cut along the axes — can only approximate it with a red staircase. The MLP (right) boundary simply rotates with the data, so it is unmoved.

This is the geometry of the bias in one picture. The tree's power on tables — carving the space one meaningful axis at a time — is the same property that makes it fragile under rotation. The MLP's rotation invariance means it cannot tell a real, meaningful column apart from an arbitrary mixture of columns; on data where the axes carry the meaning, that is a handicap, not a virtue.

Predict first

Grinsztajn ran the real version of this: take the tabular benchmark, apply a random rotation to the features, and refit every model family. Trees are not invariant; MLPs and ResNets are. Before you read the numbers, commit a guess about what the rotation does to the ranking of the models.

See it: rotation reverses the ranking

We reproduced it on a synthetic axis-aligned classification task (6 informative + 10 uninformative features; the label is a majority vote of the informative features' signs, so every feature matters on its own axis — trees' home turf). We fit each model on the original features and on a randomly-rotated copy (same Q on train and test), mean of 5 seeds.

ModelOriginalRotatedChangeInvariant?
Decision tree0.9870.747−0.240no
GBT (HistGB)0.9970.824−0.173no
Random forest0.9940.812−0.182no
MLP (256×256)0.8620.869+0.008≈ yes

Read the two columns as a story. On the original basis the tree beats the MLP by +0.126 — the axes carry the signal and the tree exploits them. After a random rotation the tree collapses to 0.747 while the MLP barely moves (+0.008, the hallmark of invariance), and now the MLP wins by −0.122. The ranking has reversed. Nothing about the information content changed — a rotation is a lossless, invertible transform — only the alignment between the signal and the axes was destroyed, and that alignment was the tree's entire advantage. Grinsztajn's conclusion: because tabular data is not invariant to rotation, a good learner should not be either.

Why does rotation invariance also make MLPs suffer more from junk features? Ng (2004) proved that any rotationally-invariant learner has a worst-case sample complexity that grows at least linearly in the number of uninformative features: to ignore a junk column it must first recover the original orientation and only then decide which directions are useless — and rotation is exactly what erases that orientation. A tree just never splits on the junk axis. That link — rotation invariance ⟶ fragility to uninformative features — is the bridge to Lesson 027.

Explain it back

This is the load-bearing idea — put it in your own words before moving on.

Thesis bridge. Rotation is the bias that speaks most directly to the relational thesis, and it cuts both ways. Against a naïve version of the bet: a relational model that ends by feeding a flattened row vector into an MLP head inherits rotation invariance and will be handicapped on exactly the tabular signal a tree exploits — so "just add a GNN then an MLP" is not a free win. For the bet: the reason rotation invariance is bad is that columns carry individual meaning — age, balance, store_id — and a good model should respect that structure rather than blend it away. A relational database is the maximal version of "the structure is meaningful": not just which column, but which entity, which foreign key, which event sequence. The same instinct that says "don't rotate the axes of a table" says "don't dissolve the schema of a database." Grinsztajn's fix on flat tables — Somepalli's SAINT and Gorishniy's FT-Transformer add a per-feature embedding that breaks rotation invariance and recovers much of the gap — is a small echo of the bigger move the thesis makes: give the model back the structure the naïve representation threw away.

Where this is subtle (know this too)

  • The invariance is not perfectly exact in practice. A raw MLP is invariant; a pipeline that per-feature-standardizes after rotation, plus finite training and random init, makes it only approximately invariant — hence the MLP's +0.008 wobble rather than a flat zero. The point is the contrast: ±0.01 for the MLP vs −0.24 for the tree.
  • Rotation is information-preserving. Q is invertible, so no signal is lost — the accuracy drop is purely the model failing to use the (still-present) signal in a scrambled basis. This is why the experiment isolates a bias, not a change in difficulty.
  • Invariance is not always bad. On images, rotation/translation invariance is a virtue (a cat is a cat, rotated) — which is why CNNs bake it in. Tabular data is the opposite regime: there is a privileged natural basis (the columns), so invariance to it destroys information about which feature is which.
  • Breaking invariance is how neural tabular models catch up. Numerical-feature embeddings (FT-Transformer, SAINT) break rotation invariance and are a large part of why modern tabular nets narrow the gap — a preview of Year 2. The embedding, not extra depth, is doing the work.
  • This is one of three biases. Rotation compounds with smoothness (L025) and uninformative features (L027), and the last two are linked by Ng's theorem. Do not over-attribute the whole gap to any single one.
