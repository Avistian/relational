Hyperparameter Search

Lesson 017 · Phase 1 · ~40 min

You now have three tunable boosters — XGBoost, LightGBM, CatBoost — each with a fistful of knobs (max_depth, learning_rate, n_estimators, subsample, regularization…). In L014 the difference between a default XGBoost and a tuned one was the whole story: 0.883 → 0.896 PR-AUC. That gain lives entirely in how you search the hyperparameter space. This lesson's single skill: explain why random search beats grid search under a fixed budget when only a few hyperparameters matter, and run RandomizedSearchCV without fooling yourself about the result.

One sentence. Given the same number of trials, grid search wastes most of them re-testing the same values of the knobs that don't matter, while random search spends every trial on a fresh value of the knob that does — so random search finds better settings as soon as your search space has a few useless dimensions (Bergstra & Bengio 2012).

The setup: a search over configurations

A hyperparameter is a setting you fix before training (unlike a weight, which is learned). Tuning is a search: pick a configuration, estimate its quality with cross-validation (the L003/L004 machinery — never the test set), keep the best. Two things define a search: the space (which knobs, over what ranges) and the budget (how many configurations you can afford to CV-evaluate). Compute is the budget — 27 configs × 5 folds is 135 model fits — so the real question is: where do you spend your trials?

Grid search and its curse

Grid search picks a handful of values per knob and evaluates the full Cartesian product. Three values each for three knobs is 3×3×3 = 27 configs; add a fourth knob and it is 81. The cost is exponential in the number of knobs — budget = kd for k values on each of d axes. To stay inside a budget as d grows, you are forced to shrink k until each axis has only 2–3 values. Worse, grid search spends the same dense effort on every axis, including the ones that barely change the score.

The insight: low effective dimensionality

Bergstra & Bengio's 2012 observation is that in practice only a few hyperparameters actually matter, and which few differs by dataset. The objective has a low effective dimensionality. Picture a 2-D space where the horizontal knob strongly changes the score (a sharp peak) and the vertical knob is nearly irrelevant (flat). Watch what an equal budget buys each strategy — the dashed lines project every trial down onto the important axis:

A grid of n points is a √n × √n lattice, so it only ever tries √n distinct values on the important axis — every column of the lattice re-tests the same important value at a different useless value. Random search draws n independent points, so it tries ≈ n distinct values on the important axis and samples the peak far more densely. Same budget; random covers the dimension that matters much better.

On our data — first the mechanism, honestly

We built a synthetic surface with a sharp peak on one important knob and near-flat useless knobs, then ran grid vs random at an equal budget (best value found, 1.000 = the true optimum; random averaged over 200 draws):

Read this honestly, because it is the whole lesson. In 2-D a lucky grid actually wins — 3 evenly-spaced values happen to straddle a centered peak, and random's scatter can miss it. The random-search advantage is not a universal law in low dimensions. It appears, and then dominates, as the number of knobs grows: at 5–8 dimensions the budget forces grid down to 2 values per axis, its resolution on the important knob collapses, and it misses the peak entirely (0.076) while random still nails it (0.967). Fix the budget at 27 and add useless dimensions one at a time and you can watch the crossover:

Random search is roughly dimension-agnostic: it keeps ≈27 distinct tries on the important axis no matter how many useless axes you bolt on. That is why it is the right default whenever you are tuning more than two or three knobs — which is always, with modern boosters.

Running it: RandomizedSearchCV

In scikit-learn you hand RandomizedSearchCV a dictionary of distributions (not fixed lists) and an n_iter budget; it draws that many configurations, CV-scores each, and keeps the best. Two practical rules:

  • Use the right distribution. For a rate that spans orders of magnitude (learning_rate from 0.01 to 0.3) sample log-uniform (scipy.stats.loguniform), not uniform — otherwise 90% of draws land above 0.03. Integer knobs use randint; fractions like subsample use uniform(0.6, 0.4) (a range of 0.6–1.0).
  • Budget is n_iter. More is better with diminishing returns; you control cost directly instead of it exploding with the number of knobs.

On real data — tuning XGBoost on credit_g, same relkit 5-fold PR-AUC as L014, at an equal budget of 27 (grid = 3×3×3 over three knobs; random = 27 draws over five knobs, two of them continuous):

StrategyBudgetcredit_g CV PR-AUC
XGBoost default (no search)10.883
GridSearchCV (3 knobs, 3 values each)270.891
RandomizedSearchCV (5 knobs, log-uniform rate)270.901

Same budget, and random search wins 0.901 vs 0.891 — matching Random Forest's 0.901 from L012 and clearing the tuned-XGBoost 0.896 from L014. It got there by tuning two extra knobs (subsample, colsample_bytree) and sampling learning_rate log-uniformly — coverage the 27-cell grid could not afford.

Don't fool yourself: the search score is optimistic

best_score_ is the CV score of the winner of a search over 27 candidates. Picking the max of 27 noisy estimates is itself a source of optimism — exactly the selection bias you met in Lesson 004. The honest number comes from nested CV: an outer loop holds out data the entire search never touches.

RandomizedSearchCV best_score_ = 0.901 (optimistic) · nested CV = 0.895 (honest) · gap +0.006

Here the gap is small (+0.006) because the search is modest, but it is always ≥ 0 and grows with the number of candidates. Rule: tune inside the training folds, and report a number that came from data the tuning never saw — a held-out test set or a nested-CV estimate. Never quote best_score_ as your final performance.

Beyond random search (a preview, not today's skill)

Random search is the baseline that is surprisingly hard to beat — but two ideas do, given enough budget:

  • Bayesian optimization (e.g. Optuna, scikit-optimize) builds a model of "config → score" and proposes the next config where improvement looks likely, instead of drawing blind. Wins when each evaluation is expensive.
  • Successive halving / Hyperband (sklearn's HalvingRandomSearchCV) starts many configs on a small budget (few trees / few rows), kills the losers early, and pours the saved compute into the survivors.

All of them still need an honest held-out estimate, and all of them are compared against random search — so random search remains the number you must report first.

Thesis bridge. The undervaluation thesis stands or falls on fair baselines. "Our relational model beats XGBoost" means nothing if the XGBoost got its defaults while your model got a 500-trial Bayesian sweep. A fair comparison gives every model — the GBDT baseline and the fancy relational one — the same tuning budget under the same CV protocol, and reports a nested / held-out number. This is not pedantry: the Year 1 Q3 benchmark-literacy lectures and the Y3 "fair tuning budget on the entity graph" lecture both hinge on it. Random search is the equalizer, because it lets you hand every model the identical budget regardless of how many knobs each one has.

Where this breaks (know this too)

  • Very low dimension. With one or two knobs, a well-placed grid can match or beat random (our 2-D rows) — the advantage is about many knobs, most useless.
  • Discrete, tiny spaces. If a knob has three legal values, just grid it — random sampling of three options wastes draws on duplicates.
  • Bad ranges beat any strategy. Search is only as good as the space you define; a log-uniform learning_rate over the wrong decade finds nothing. Sensible ranges > clever search.
  • Optimism is unavoidable without a held-out set. No search strategy removes selection bias; only an outer split (nested CV / test set) gives an honest number.
