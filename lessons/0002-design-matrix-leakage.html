Design Matrix & Leakage

Lesson 002 · Phase 1 · ~45 min

Lesson 001 named the contract: tabular learners eat a flat matrix. This lesson makes that contract precise, catalogs the ways joins cheat, and gives you a checklist you will reuse through Year 6.

The design matrix

A design matrix X has shape (n, d): n rows (one per training example) and d columns (features). The label vector y has length n — row i of X pairs with y[i]. That row alignment is the whole API. 

customers.csv  +  orders.csv  ──join/aggregate──▶  X (n×d)  +  y (n)

XGBoost, sklearn, and neural tabular models never see customers or orders as separate tables. They see only X and y. See the leakage cheat sheet.

Row alignment pitfalls

Leakage often enters before any “future date” mistake — at the row level:

  • Duplicate keys after join. Joining orders to customers without GROUP BY customer_id produces one row per order, not per customer. Labels defined at customer level no longer align with feature rows.
  • Left vs inner join. Inner joins drop customers with zero orders, changing the population y was defined on. That is sample bias, not always leakage — but it invalidates comparison to a full cohort.
  • Train and test built separately. Fitting a scaler or target encoder on the full dataset before the split leaks test statistics into train features. Pipelines must fit transforms on train only.

Point-in-time prediction

Every leakage audit starts with one question: what did we know at decision time t? For churn, t might be “last day of the billing period.” For credit, t is application time. Features may use data strictly ≤ t. Labels describe outcomes strictly after t (e.g. cancel within 30 days). Anything from the label window in the features is leakage. 

timeline:  ─── past features ───| t |─── label window ───▶
allowed:     order_date ≤ t          ^     cancel_date ∈ (t, t+30d]

Join sketch

Task: predict whether customer c churns in the 30 days after date t.

customers(customer_id, signup_date, country) orders(order_id, customer_id, order_date, amount) Feature ideas built by joining on customer_id: A) total_spend = SUM(amount) -- all orders, no date filter B) spend_30d = SUM(amount WHERE order_date ∈ (t−30d, t]) C) future_orders = COUNT(*) WHERE order_date > t D) country = customers.country E) days_since_last_order = t − MAX(order_date WHERE order_date ≤ t) F) avg_order_after_t = AVG(amount WHERE order_date > t)

Find the leakage

Which features smuggle information that would not exist at prediction time t?

Verdict table

FeatureVerdictWhy
A — total_spend Leak Sums orders after t unless explicitly filtered
B — spend_30d Safe Window ends at t; only past behavior
C — future_orders Leak Uses only post-t orders by construction
D — country Safe Static attribute known at signup (assuming no retroactive edits)
E — days_since_last_order Safe MAX restricted to orders ≤ t
F — avg_order_after_t Leak Directly averages future transactions

Four leakage patterns to memorize

  1. Temporal bleed. Future timestamps in features (A, C, F above). Most common in relational data because SQL makes full-table aggregates easy.
  2. Label bleed. Features derived from the outcome column or its proxies (e.g. “reason for cancellation” encoded before the model runs at t).
  3. Population bleed. Filtering or imputing using test-set statistics; target encoding fit on all data; duplicate rows that cross split boundaries.
  4. Pipeline bleed. Feature jobs that re-run on all history nightly and back-fill values into training snapshots — common in production, invisible in notebooks.

Why this matters for RDL

Manual feature engineering hides leakage in SQL no one re-reads. Relational models (GNNs on a REG) still need temporal splits and point-in-time features — the graph does not magically fix a leaky label horizon. A message-passing layer can propagate information across edges; without time masks, it propagates future information too. Evaluation discipline is non-negotiable in Phase 1 and stays non-negotiable in Year 4.

Your homework pipelines already use grouped CV — extend that instinct to joins. Before any benchmark claim, ask: “Could this feature be computed from a cron job that only had access to data ≤ t?” If not, fix the SQL first, then compare models.

Primary reading (today)
