Design Matrix & Leakage

Lesson 002 · Phase 1 · ~20 min

Lesson 001 named the contract: tabular learners eat a flat matrix. This lesson makes that contract precise and shows where joins most often cheat.

The design matrix

A design matrix X has shape (n, d): n rows (one per training example) and d columns (features). The label vector y has length n — row i of X pairs with y[i]. That row alignment is the whole API. 

customers.csv  +  orders.csv  ──join/aggregate──▶  X (n×d)  +  y (n)

XGBoost, sklearn, and neural tabular models never see customers or orders as separate tables. They see only X and y. See the leakage cheat sheet.

Join sketch

Task: predict whether customer c churns in the 30 days after date t.

customers(customer_id, signup_date, country) orders(order_id, customer_id, order_date, amount) Feature ideas built by joining on customer_id: A) total_spend = SUM(amount) -- all orders, no date filter B) spend_30d = SUM(amount WHERE order_date ∈ (t−30d, t]) C) future_orders = COUNT(*) WHERE order_date > t D) country = customers.country

Why this matters for RDL

Manual feature engineering hides leakage in SQL no one re-reads. Relational models (GNNs on a REG) still need temporal splits and point-in-time features — the graph does not magically fix a leaky label horizon. Evaluation discipline is non-negotiable in Phase 1 and stays non-negotiable in Year 4.

Primary reading (today)

Chip Huyen — Designing Machine Learning Systems, Chapters 2–3 (data fundamentals and feature engineering). Focus on how train/valid/test splits and feature pipelines interact — stop before deployment chapters.
