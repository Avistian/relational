Relational Data Without RDL

Lesson 034 · Phase 1 · Q4 · ~45 min · primary reading: Kimball & Ross, dimensional modeling — “Fact Tables and Dimension Tables”

Last lesson (L033) ended on a phrase we now have to make literal: “the returns moved across the join.” On a single flat table, hand-crafted features hit diminishing then negative returns almost immediately — yet the whole time we have been assuming that flat table simply exists. It does not. Real data lives in a relational database: several tables, linked by keys, at different grains. Before you can fit any of the models from Q1–Q3, someone has to flatten that database into one design matrix — and that flattening is a modelling choice with a cost. This lesson teaches you to read a small relational schema and to write the join-and-aggregate that turns it into a single leak-free training table. This is the entire single-table paradigm the thesis critiques, built by hand, so that L035 (“What joins destroy”, next) can measure exactly what it throws away.

Its single skill: read a three-table schema (a fact table plus its dimensions, with primary and foreign keys) and write the join + point-in-time aggregation that flattens it into one design matrix at a chosen entity grain — keeping every aggregate strictly to information available before the prediction time.

One sentence. A relational database stores entities in separate tables linked by foreign keys; to feed a standard model you pick a grain (one row per entity at one prediction time) and flatten the neighbouring tables into columns by joining and then aggregating — and every aggregate must use only rows dated before the prediction time, or you leak the future.

First, the vocabulary (nothing here is assumed)

This lesson introduces the language of relational data. Each term is defined here in plain language, from first principles, before we rely on it — assume you are meeting each one fresh, even if an earlier lesson brushed past it.

Relational database. A collection of tables, each a grid of rows and named columns. Rather than duplicate information, related facts are split across tables and re-linked by shared key columns. “Relational” refers to each table being a mathematical relation (a set of rows), not to the links between them — though the links are what make it useful here.
Entity. A thing the data is about — a customer, a product, an order. In ML we usually predict for an entity (“will this customer churn?”), so one entity type becomes the unit our training rows describe.
Primary key (PK). The column (or columns) that uniquely identifies each row of a table — customer_id in a customers table. No two rows share a PK value; it is the row's stable name.
Foreign key (FK). A column in one table that stores the primary-key value of a row in another table, creating a link. orders.customer_id is a foreign key into customers.customer_id: it says “this order belongs to that customer.” Foreign keys are the edges of the database — and, later in the thesis, the edges of a graph.
Fact (event) table vs. dimension table. In dimensional modelling (Kimball & Ross), a fact table records events / measurements — an order, a payment, a click — usually with a timestamp and numeric measures, and holds foreign keys pointing out to dimension tables, which describe the entities / context (the customer, the product, the store). Facts are long and grow over time; dimensions are comparatively static descriptions.
Star schema. The classic dimensional layout: one central fact table surrounded by its dimension tables, joined by PK/FK links — drawn, it looks like a star. It is the standard way analytics data is organised, and the mental model you bring to any new database: find the events, find the entities they point to.
Grain. What one row of a table means — its unit of observation. order_items has grain “one line item”; orders has grain “one order”; customers has grain “one customer.” Choosing the grain of your training table (“one row per customer at prediction time”) is the first and most consequential modelling decision, because it fixes what a prediction is about.
Cardinality of a relationship (one-to-many). How many rows on each side a link connects. One customer has many orders (1 : ∞); one order has many line items. A one-to-many link is exactly what forces aggregation: to describe the “one” side with the “many” side, you must summarise the many into a fixed set of numbers.
Join. The operation that combines rows from two tables wherever a FK matches a PK, producing rows that carry columns from both. Joining orders to customers on customer_id attaches each customer's attributes to each of their orders.
Aggregation. Collapsing many rows into one summary number per group — COUNT, SUM, AVG, MAX, … Because a customer has many orders but a model wants one row per customer, aggregation is how the “many” becomes columns on the “one.” Every aggregate is lossy by construction: AVG(amount) keeps the mean and forgets the individual amounts.
Design matrix (flat table). The single rectangular table a standard model consumes: one row per training example, one column per feature, one target. Producing it from a database is the flatten. Everything in Q1–Q3 assumed this matrix was handed to you; this lesson builds it.
Point-in-time (PIT) correctness. Each training row is tied to a prediction time \(t\); every feature for that row must be computable from data available at or before \(t\). An aggregate that sums orders placed after \(t\) uses the future — the label leakage of L002 / L021 / L022, now committed at the feature-construction step.

The schema you will flatten

Meet the toy database used for the rest of the lesson (and the lab): three tables — a customers dimension and two fact tables, orders and its detail order_items. Use the buttons to trace each foreign-key link and read its cardinality. The grain of each table is printed beneath it.

Read the structure the way an analyst does: orders is the event stream (each row a purchase, with a timestamp), pointing back to the customers who made it; order_items is the finer grain beneath each order (each row one product line). The links are one-to-many in the fact→dimension direction: one customer, many orders; one order, many items. That “many” is the crux — a model that predicts per customer cannot ingest a variable-length list of orders. It needs a single row.

The ML row is “one entity at one time” — and the flatten fills it

Suppose the task is: predict whether a customer churns in the 30 days after a cutoff date t. Then the natural training grain is one row per customer, as of t. The label comes from the future window (did they churn after t?); the features must summarise everything known about that customer up to t. But the customer's behaviour lives in orders and order_items, at a finer grain. Bridging that gap is the flatten: join the neighbour tables in, then aggregate them into columns at the customer grain.

Writing the flatten: join, then aggregate — leak-free

Here is the skill in code. We want, for each customer and cutoff t, a leak-free summary of their orders strictly before t. In SQL the point-in-time filter is a single WHERE clause on the timestamp; the aggregation is a GROUP BY the entity key: 

-- one leak-free feature row per customer, as of cutoff :t
SELECT
    c.customer_id,
    c.segment,
    COUNT(o.order_id)              AS n_orders_before_t,
    COALESCE(SUM(o.amount), 0)     AS total_spend_before_t,
    COALESCE(AVG(o.amount), 0)     AS avg_basket_before_t,
    MAX(o.order_ts)               AS last_order_ts_before_t
FROM customers c
LEFT JOIN orders o
       ON o.customer_id = c.customer_id
      AND o.order_ts < :t          -- POINT-IN-TIME: only the past
GROUP BY c.customer_id, c.segment;

The same operation in pandas is a filter, a groupby, and a merge back onto the entity table: 

# orders_before = orders placed strictly before the cutoff t
past = orders[orders.order_ts < t]
agg  = (past.groupby("customer_id")
             .agg(n_orders_before_t=("order_id", "count"),
                  total_spend_before_t=("amount", "sum"),
                  avg_basket_before_t=("amount", "mean")))
flat = customers.merge(agg, on="customer_id", how="left").fillna(0)

Three things make this correct, and each is a place people slip:

  • The grain is fixed by the GROUP BY / groupby key. Grouping by customer_id guarantees exactly one output row per customer — the design-matrix grain. Forget it and you get one row per order (the fact grain), and your “per-customer” model is quietly training on duplicated, order-level rows.
  • The LEFT JOIN keeps customers with zero past orders. A plain (inner) join would silently drop new customers who have not ordered yet — a biased training set. The COALESCE/fillna(0) then encodes “no orders” as a count of 0, not a missing value you have to explain later (L006).
  • The order_ts < :t filter is the point-in-time guard. It must live in the join/aggregation itself, not be bolted on afterwards. Drop it and every aggregate peeks at orders the customer placed after the prediction time — future information baked straight into a feature.

This is the L002 lesson, now at the feature-engineering step of a relational flatten. Notice that the schema itself gives you no protection — SQL will happily average future orders. Point-in-time correctness is a discipline you impose, re-impose on every aggregate, and re-impose again the next time the same database is flattened for a different task. That “re-impose every time” is a running cost we will return to.

Explain it back

This is the load-bearing idea — put it in your own words before moving on.

Thesis bridge. Everything in this lesson is the single-table paradigm the thesis argues is undervalued-against. Three observations set up the bet. (1) The flatten is a hand-built pipeline, not a given. Someone chooses the grain, writes each join, picks each aggregate (COUNT, AVG, MAX), and guards each with a point-in-time filter — the design matrix Q1–Q3 took for granted is the output of exactly this human labour. (2) Each aggregate is a hand-designed Deep Feature Synthesis primitive. “Average basket over the last 90 days,” “number of prior orders,” “days since last order” — these are the relational features L033 said still pay, and here you see they are produced by walking foreign keys and summarising. (3) The structure is being deliberately discarded. The one-to-many link, the order in which baskets arrived, which products were bought, the identity of each event — the aggregation throws all of it away to make a fixed-width row. L035 (next lesson) quantifies that loss; Year 3–4 keep the same PK/FK edges as a graph and let a model learn the aggregates end-to-end instead of you hand-writing them. The relational thesis, in one line: don't flatten the database into columns — learn over its structure directly.

Where this is subtle (know this too)

  • Choosing the grain is choosing the question. “One row per customer at t” and “one row per customer per month” are different design matrices with different labels and different leakage surfaces. The grain is not cosmetic — it defines what a prediction is.
  • Aggregation is lossy on purpose, and that is the whole point of L035. The mean of a basket is a deliberate compression. It buys the fixed-width row a model needs, at the price of the cardinality, order, and identity of the underlying events. “Add more aggregates” recovers some of it and never all of it.
  • Multi-hop paths multiply the work. Reaching product categories from a customer is customer → orders → order_items → products → categories — several joins, each aggregated. Every hop is code to write, tune, and keep leak-free; the deeper the path, the more the manual flatten strains.
  • The same join is redone per task. A churn model, a lifetime-value model, and a fraud model over the same database each rebuild their own flatten with their own cutoffs and aggregates. The relational structure is re-flattened, re-audited, and re-leaked-checked every time — a human cost that scales with the number of tasks, which the thesis argues a learned relational model amortises.
  • Star vs. snowflake vs. raw OLTP. A clean Kimball star is the idealised analytics shape; a live transactional (OLTP) database is more normalised and messier. The reading skill is the same — find the fact/event tables, find the dimensions they point to, note the grains and keys — but the real flatten is rarely as tidy as the toy schema above.
