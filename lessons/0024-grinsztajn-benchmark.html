The Grinsztajn Benchmark

Lesson 024 · Phase 1 · Q3 · ~40 min · core paper: Grinsztajn, Oyallon & Varoquaux 2022

The last three lessons made a single comparison honest: the right split (L021), leak-free features (L022), and a significance test on the gap (L023). Now we zoom out from one comparison to a whole benchmark: dozens of datasets, several model families, and a protocol designed so the headline — "tree ensembles still beat deep learning on typical tabular data" — is a defensible finding rather than a lucky configuration. This lesson opens the Grinsztajn arc (024–027): here we build and read the benchmark; L025–L027 dissect the three inductive biases that explain the result (you met them in preview in L019).

Its single skill: read and reproduce the Grinsztajn benchmark protocol — compare models across a random-search hyperparameter budget (not one tuned number), normalize test scores per dataset so they can be aggregated, and know the dataset-selection criteria that make the tree-vs-DL verdict fair.

One sentence. A benchmark's verdict is only as trustworthy as its dataset curation and its tuning protocol: Grinsztajn compares models not at a single tuned point but along the whole random-search budget curve, on a suite of datasets chosen by explicit rules, and normalized so no single dataset dominates the average.

Why one tuned number lies

Suppose you read "FT-Transformer: 0.82, XGBoost: 0.81." Which is better? You cannot tell, because a single number hides the question that decides everything: how much tuning did each model get? A neural net with a hundred sensitive knobs might need 200 random-search iterations to reach 0.82, while a GBT reaches 0.81 on its defaults. Report only the tuned peak and you flatter the model that is hardest to tune — and you hide the fact that one of them is far cheaper to deploy.

Grinsztajn's fix (their §4) is to stop reporting a point and report a curve: for each model and dataset, draw many random hyperparameter configurations, and plot the expected test score of the best-so-far configuration as a function of the number of search iterations. Two properties make this fair: it shows both the default quality (budget = 1) and the tuned ceiling (large budget), and — because it averages over random orderings of the draws — it is a smooth, reproducible object you can actually compare. Before any of that, though, comes the quieter half of fairness: the datasets.

Fairness starts with the datasets (§3)

If you get to choose the datasets, you can "prove" almost anything. So the first defence against a dataset-selection artifact is an explicit, pre-registered set of inclusion criteria, applied to a large pool before any model is run. Grinsztajn filters OpenML down to a suite of medium-sized, heterogeneous, real, non-trivial tabular datasets. Click through the funnel to see each rule and why it is there (the counts are illustrative; the criteria are the point).

Two criteria deserve emphasis because they are exactly where a sloppy benchmark cheats. "Heterogeneous columns" excludes data whose columns are really pixels or sensor channels — that is perceptual data in disguise, where neural nets have a natural home-field advantage, and including it would smuggle a non-tabular win into a tabular benchmark. "Not too easy" removes datasets where a default linear model already ties a tuned GBT: such datasets cannot discriminate methods, so they only add noise to the aggregate. Curation is not housekeeping — it is half the experiment.

Two honest readings. First, tuning helps the neural net far more than the GBT: the MLP climbs from 0.729 to 0.770 as the budget grows, while the GBT barely moves — the gap narrows (+0.062 → +0.015) but never closes. That is the shape of Grinsztajn's Figure 1 across the whole suite: GBTs are both a stronger default and a stronger ceiling. Second — a real small-data artifact worth naming — the GBT curve here even dips slightly at large budgets: with only 200 validation rows, picking the best-of-many configs starts to overfit the validation set (an echo of L004's selection bias and L017's tuning honesty). The full benchmark avoids this by using larger, medium-regime datasets — which is exactly why "not too small" is a selection criterion.

Why you cannot just average accuracies

To turn per-dataset curves into one benchmark number you must average across datasets — but an 80% on an easy dataset and an 80% on a hard one are not commensurable (the same worry you met in L023, where Demšar refuses to average raw accuracies). Grinsztajn's answer is an affine per-dataset normalization: on each dataset, rescale test scores so the worst model maps to 0 and the best to 1, then average those normalized curves.

normalized(score) = ( score − worstdataset ) / ( bestdataset − worstdataset )

Flip the widget above to Normalized: on credit-g the MLP's untuned point (the worst model seen) maps to 0.00 and the GBT's peak to 1.00, and the same ordering survives. Averaging these bounded, per-dataset scores stops one high-variance dataset from hijacking the mean — the aggregation step that makes a 45-dataset headline meaningful.

What the benchmark actually concludes

On this suite, under this protocol, tree ensembles (XGBoost, GBTs, Random Forests) outperform the neural-net families (MLP, ResNet, FT-Transformer, SAINT) at every tuning budget, and the advantage is largest at small budgets (trees are better defaults). Crucially, Grinsztajn does not stop at "trees win" — the paper's real contribution is why, framed as three inductive biases you previewed in L019 and will dissect next:

  • L025 — smoothness. Neural nets are biased toward smooth functions; tabular targets are often irregular, so MLPs over-smooth them.
  • L026 — rotation. MLPs are rotationally invariant, but tabular features are individually meaningful (a column is not an arbitrary linear mix); trees respect that orientation.
  • L027 — uninformative features. Real tables carry junk columns; trees select against them, MLPs are hurt by them.

The benchmark is the measuring instrument; the biases are the mechanism. Getting the instrument right — datasets and protocol — is what earns you the right to draw a mechanistic conclusion at all.

Thesis bridge. This is the honest bar the relational thesis must clear, made concrete. When an RDL model is compared to a GBT, the fair report is not a single tuned peak — it is a budget curve against a GBT that is a strong default, over a curated set of tasks, with normalized aggregation. RelBench exists precisely to be that instrument for the relational setting (many tasks, fixed protocol, point-in-time splits), just as Grinsztajn's suite is for single tables. And note the boundary the thesis actually attacks: Grinsztajn's whole benchmark lives inside the single-table world — it curates away relational structure (heterogeneous columns, one flat matrix). The tree-vs-DL contest is a fight over how to model a flattened table; the thesis's claim (C1) is that the flattening itself discarded signal before either contestant sees the data. Beating a GBT on Grinsztajn's terms is table stakes; the bet is on a different table.

Where this is subtle (know this too)

  • The verdict is regime-specific. Grinsztajn studies medium-sized (≈10K training rows), heterogeneous data. Larger data, homogeneous/perceptual columns, or the modern tabular nets of Year 2 (TabM, TabPFN, TabICL) narrow or flip it — the benchmark's scope is its claim.
  • A budget curve is not free. Running hundreds of configs per model per dataset is expensive; the payoff is a comparison that can't be gamed by tuning one side harder.
  • Selection criteria are choices, not laws. "Not too easy" and "heterogeneous" are defensible but debatable; a critic can argue a criterion tilts the field. That the rules are explicit is what makes the argument possible at all.
  • Default ≠ ceiling. A model can be a poor default and a strong ceiling (the MLP here) or a strong default and a modest ceiling (the GBT). One tuned number erases this distinction; the curve keeps it.
  • This is a single-dataset reproduction. The credit-g curve shows the protocol, not the full 45-dataset result — for that, read the paper and, later, run the repo.
