Train / Valid / Test & Cross-Validation

Lesson 003 · Phase 1 · ~40 min

Lesson 002 taught you to audit features at prediction time t. This lesson teaches where those features may be used: which rows train the model, which rows tune it, and which rows earn a final score. Get this wrong and even a leak-free join sketch produces a lying metric.

Why hold out data at all?

Training and evaluating on the same rows is a methodological mistake. A model that memorizes labels would score perfectly and generalize nowhere. The scikit-learn cross-validation guide states this plainly: hold out part of the data as a test set before you declare victory.

Your thesis will be judged on generalization — on RelBench tasks, on temporal splits, against tuned GBDT baselines. Split discipline is how you keep those comparisons honest.

Three sets, three jobs

SetJobRule of thumb
Train Fit model parameters (tree splits, neural weights) Use freely during development
Validation Choose hyperparameters, early stopping, feature sets Peek many times; never confuse with test
Test Final estimate of out-of-sample performance Touch once — then the estimate is spent

Hyperparameter tuning on the test set is the classic mistake: you keep tweaking until the test metric looks good, so test knowledge leaks into the model. The fix is either a dedicated validation set or cross-validation (CV) on the training portion — CV rotates which rows play validation so you do not waste data on a single fixed validation slice.

Holdout split: train_test_split

The simplest pattern: one random partition into train and test. In sklearn: 

from sklearn.model_selection import train_test_split

X_train, X_test, y_train, y_test = train_test_split(
    X, y,
    test_size=0.2,      # 20% held out
    stratify=y,         # preserve class ratio in both sets
    random_state=42,    # reproducible split
)

stratify=y matters when labels are imbalanced (churn, fraud, rare disease). Without it, a lucky random draw can put almost all positives in train and none in test — or vice versa — making accuracy and PR-AUC meaningless. See the CV cheat sheet.

Worked example: 10% positive churn

10,000 customers, 1,000 churners. A 80/20 split without stratification might leave only 150 churners in test (unlucky) or 250 (lucky). You cannot compare two models if the test set's difficulty changes every seed. Stratification keeps ~10% positives in both train and test.

K-fold cross-validation

Split the training data into k folds. For each fold:

  1. Train on k−1 folds
  2. Score on the held-out fold
  3. Average the k scores
from sklearn.model_selection import cross_val_score
from sklearn.linear_model import LogisticRegression

clf = LogisticRegression(max_iter=1000, random_state=42)
scores = cross_val_score(clf, X_train, y_train, cv=5)
print(f"{scores.mean():.3f} ± {scores.std():.3f}")

For classifiers, cross_val_score(..., cv=5) uses StratifiedKFold by default — each fold preserves the global class ratio. Report mean ± std across folds; high variance means either too little data or a noisy task.

Lesson 002 meets Lesson 003: pipelines

Lesson 002 flagged pipeline bleed: fitting a scaler or target encoder on all of X before splitting leaks test statistics into train features. sklearn's rule: preprocessing must be fit on training data only, then applied to held-out rows. 

from sklearn.pipeline import make_pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.linear_model import LogisticRegression

pipe = make_pipeline(StandardScaler(), LogisticRegression(max_iter=1000))
scores = cross_val_score(pipe, X_train, y_train, cv=5)

Inside each CV fold, the scaler fits on that fold's train slice and transforms only that fold's validation slice. This is the pattern your homework pipelines should already follow — if not, fix it before any benchmark claim.

What this lesson does not cover yet

You already use grouped CV in production-style pipelines. Standard K-fold assumes rows are i.i.d. — independent draws from the same distribution. sklearn's docs warn explicitly: if samples have group structure (multiple rows per person, per store, per patient), use group-wise CV instead. That is Lesson 004 (Grouped & nested CV).

Similarly, random splits assume no time ordering. Production churn and RelBench tasks need temporal splits (Lesson 021). Random stratified CV is the right tool for i.i.d. single-table baselines in Phase 1 — not the final word on relational evaluation.

scikit-learn (1.9 docs): “If one knows that the samples have been generated using a time-dependent process, it is safer to use a time-series aware cross-validation scheme.” Bookmark that sentence — you will cite it in Year 4.

Four split mistakes to catch

  1. Test-set tuning. Running grid search on X_test until metrics improve. Fix: CV on train, or train/val/test with val for search.
  2. Preprocessing bleed. Global fit_transform(X) before split. Fix: Pipeline inside CV.
  3. Unstratified rare labels. Lucky/unlucky class ratios in folds. Fix: stratify=y and StratifiedKFold.
  4. Wrong independence assumption. Same customer in train and test. Fix: grouped splits — next lesson.
