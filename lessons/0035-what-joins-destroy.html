What Joins Destroy

Lesson 035 · Phase 1 · Q4 · ~40 min · primary reading: Fey et al., “Relational Deep Learning” (2024), §1–2 (★ preview)

Last lesson (L034) you built the flatten: you took a three-table schema and hand-wrote the join + point-in-time aggregation that turns it into one design matrix. You also saw, in passing, that aggregation is lossy by construction — AVG keeps a mean and forgets the individual baskets. This lesson makes that loss precise: it enumerates exactly which structural information a flatten-then-aggregate pipeline throws away, and shows a case where the loss is total — two genuinely different entities collapsing to the same feature row. This is the intellectual pivot of Year 1: it is the moment the single-table paradigm stops looking like “just how data is” and starts looking like a choice with a measurable cost. Everything the thesis proposes in Years 3–6 is an attempt to avoid paying it.

Its single skill: enumerate the structural information a flatten-then-aggregate pipeline discards — cardinality, event identity, temporal order within a neighbour set, and higher-order (multi-hop) paths — and recognise an aggregation collision, where two entities with different neighbourhoods map to the same fixed-width feature vector.

One sentence. Joining and aggregating related tables into fixed-width columns is a surjective, lossy map: many different relational neighbourhoods send to the same feature row, so a model on the flat table cannot recover the discarded structure (order, identity, count, multi-hop paths) no matter how well it is tuned — which is the information Fey et al. argue relational deep learning keeps by learning over the primary-/foreign-key graph directly.

First, the vocabulary (nothing here is assumed)

This lesson leans on a few precise terms. Each is defined here from first principles before we rely on it, even if L034 or an earlier lesson brushed past it.

Lossy vs. lossless map. A transformation is lossless if you can invert it — recover the original input from the output (a random rotation of features, L026, is lossless: Q is invertible). It is lossy if you cannot: several distinct inputs produce the same output, so the original is unrecoverable. Aggregation is lossy — that is the whole subject of this lesson.
Aggregation collision. When two different inputs (here, two entities with different related rows) produce the identical aggregated feature vector. A collision is the sharpest possible proof of lossiness: the flat table literally cannot tell the two entities apart, so any model — a tuned GBDT, an MLP, anything — must assign them the same prediction.
Cardinality (of a neighbour set). How many related rows an entity has — the number of orders a customer placed, of items in a basket. A COUNT keeps it; most other aggregates (AVG, MAX) discard it.
Event / entity identity. Which specific rows an entity is linked to — which products, which merchant, which other customer. An aggregate over amounts keeps magnitudes and drops identities: SUM(amount) does not know whether the money went to milk or to wine.
Temporal order within a neighbour set. The sequence in which an entity's related events arrived. Point-in-time correctness (L034) controls which events are eligible (before t); it does not preserve their order. A rising vs. a falling spend trajectory can have identical sum/mean/max.
Higher-order / multi-hop path. A relationship reached by following more than one foreign key: customer → orders → order_items → product → category. Each hop is a separate join, and a flat aggregate over one hop cannot see two hops out — the product a customer favours is invisible to any summary of their orders alone.
Relational entity graph. Fey et al.'s reframing: treat the database as a graph with one node per row in every table and an edge for every primary-/foreign-key link. Nothing is flattened; the schema is the graph. RDL learns representations by message passing over this graph — the alternative to the flatten this lesson dissects.

The taxonomy: what a flatten-then-aggregate pipeline discards

The collision packs several distinct losses into one example. Pulling them apart gives the checklist you should run against any hand-built relational feature set — “what did this aggregate just throw away?”

Discarded structureWhat it isWhich aggregates lose itIn the collision
Cardinality How many related rows (orders, items) Everything except COUNT — AVG/MAX/SUM-of-ratios preserved here (both n=3) — but usually the first casualty
Event identity Which specific products / merchants / entities Any numeric aggregate over amounts lost — milk/bread/eggs vs. wine×3 both vanish into $90
Temporal order The sequence the events arrived in SUM, MEAN, MAX, MIN — all order-blind lost — rising vs. falling both give sum 90 / max 50
Higher-order paths Signal ≥2 hops away (customer→order→item→product→category) Any single-hop aggregate; each extra hop is a new hand-written join lost — product-level structure never enters an order-level summary

Notice the pattern: an aggregate is a projection. It maps a variable-length, structured neighbourhood down to a few fixed numbers, and projections lose the dimensions they collapse. You can recover some of a lost dimension by adding a targeted aggregate — a spend_trend (last − first) recovers order; a n_distinct_products recovers a slice of identity — but each is another hand-written, per-task, leak-checked feature, and you can never enumerate them all in advance. That “add more aggregates forever” treadmill is the cost the thesis is about.

Fey et al.'s argument, in their words

This is the ★ preview of the paper the whole mission orbits: Fey, Hu, Huang, Lenssen, Ranjan, Robinson, Ying et al., “Relational Deep Learning: Graph Representation Learning on Relational Databases” (2024). The paper opens by naming the problem exactly as you just built it: “no machine learning method is capable of learning on multiple tables interconnected by primary-foreign key relations. Current methods can only learn from a single table, so the data must first be manually joined and aggregated into a single training table, the process known as feature engineering.” It then lists five issues with that approach (§1–2):

  1. it is a manual, slow and labour-intensive process;
  2. the feature choices are essentially arbitrary and likely highly suboptimal;
  3. only a small fraction of the space of possible features can be manually explored;
  4. by forcing data into a single table, information is aggregated into lower-granularity features, thus losing out on valuable fine-grain signal — this is the collision you just saw, stated as a thesis;
  5. when the data distribution drifts, the hand-built features become obsolete and must be manually reinvented.

Fey et al. draw the analogy to pre-deep-learning computer vision, where hand-chosen filters (e.g. Gabor) fed an SVM, until deep nets learned directly on raw pixels and “skipped the feature engineering.” Their proposal is the same move for databases: view the relational database as a temporal, heterogeneous graph — a node per row, edges from primary-/foreign-key links — and let a message-passing GNN learn representations over it end-to-end, so the aggregations you hand-wrote in L034 are learned, not authored, and the fine-grain signal item (4) is never discarded in the first place.

Explain it back

This is the load-bearing idea of Year 1 — put it in your own words before moving on.

Thesis bridge. L034 showed the flatten is a choice; L035 measures its cost. The cost is not inefficiency — it is information: the join-then-aggregate map is lossy, and the collision proves the loss can be total (different entities, one feature row). Fey et al.'s bet is that this discarded structure — cardinality, identity, order, multi-hop paths — carries real predictive signal, and that a model reading the relational entity graph (rows = nodes, foreign keys = edges) can learn to use it without a human hand-writing an aggregate for each dimension of loss. This closes Year 1's conceptual arc that opened at L001 (“the single-table assumption”): the flat table was never given, it was manufactured (L034), and manufacturing it destroys structure (L035). What it does not yet show is a model actually recovering that structure to beat the honest single-table bar (the tuned GBDT + ResNet + AutoML of L028–L030) — that demonstration is the burden of the Year-1 exit essay and, for real, of Years 3–4 (message passing, the relational entity graph).

Where this is subtle (know this too)

  • Lossy is not the same as useless. Aggregates are excellent features — L009's Deep Feature Synthesis and every winning Kaggle pipeline lives on them. The claim is narrower and exact: aggregation throws away recoverable structure, and when that structure carries signal, no single-table model can get it back. “Lossy” is a statement about information, not about value.
  • You can always add another aggregate — that is the trap, not the escape. Every collision can be broken by a bespoke feature (add spend_trend, add n_distinct_products). But the space of collisions is unbounded, the features are per-task, and each must be re-derived and re-leak-checked when the schema or the task changes (issue 5). Hand-built features chase losses one at a time; a learned model over the graph addresses the class.
  • Point-in-time correctness and lossiness are different failures. L034's leak was about using the future; L035's loss is about discarding the past's structure. A pipeline can be perfectly leak-free (only pre-t orders) and still throw away the order/identity/paths of those legitimate rows. Fixing one does nothing for the other.
  • Trees don't escape it either. A gradient-boosted tree's inductive-bias advantages (L019, L025–L027) are all about how it reads the columns it is given. If the collision has already erased the signal from those columns, the tree's robustness to junk features and its axis-aligned splits are irrelevant — there is nothing left in the row to split on. This is why “just use XGBoost” is not an answer to the flatten's loss.
  • The graph reframing is not free lunch, yet. Keeping the PK/FK graph poses its own hard problems — time-respecting message passing, stale links, scale — which Fey et al. flag as open (and Years 3–6 tackle). L035's job is only to establish that the structure exists and is being thrown away, making it worth the trouble of keeping.
