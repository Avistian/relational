CatBoost

Lesson 016 · Phase 1 · ~40 min

XGBoost gave you the regularized gain math; LightGBM re-engineered it for speed. CatBoost (Prokhorenkova et al., NeurIPS 2018) is the third member of the GBDT family, and it earns its place by attacking a completely different problem: leakage. Specifically, two leaks that quietly inflate ordinary gradient boosting — the target leak in categorical encoding, and the subtler prediction shift in the boosting loop itself. This lesson's single skill: explain how a random permutation and "use only the past" (ordered target statistics + ordered boosting) make categorical encoding and gradient estimation leakage-free, know that CatBoost's trees are oblivious (symmetric), and confirm where CatBoost actually wins.

One sentence. CatBoost is gradient boosting that encodes each categorical value and estimates each row's gradient using only rows that come earlier in a random permutation — so a row's own label never leaks into its own features or its own gradient — grown as fast, symmetric "decision-table" trees.

The problem: greedy target statistics leak

Trees split on numeric thresholds, so a categorical column (city, user_id, product) must become a number. The most powerful encoding is a target statistic (TS): replace each category with the mean target for that category — "customers in this city churn 22% of the time." It is far stronger than one-hot for high-cardinality columns. But computed naively it leaks the label: to encode row i you average the target over all rows of that category including row i's own label. For a rare category — say a user_id seen twice — that average is dominated by the row's own y. The feature becomes a peek at the answer.

You met this exact trap in Lesson 009 (target encoding scored 0.76 on pure noise until we moved it inside a per-fold pipeline) and the leakage discipline behind it in L002 (point-in-time joins) and L004. Prokhorenkova et al. name the training-loop version of the same disease prediction shift: standard boosting estimates each round's gradients on the same rows used to build the current model, so the gradient for a row is computed by a model that has already seen that row. The estimates are biased, and the model overfits in a way ordinary CV can hide.

The fix: use only the past (a random permutation)

CatBoost's core idea is disarmingly simple — impose an artificial "time" by drawing a random permutation of the rows, then compute everything for row i using only the rows before it in that permutation (its prefix). A row can never see its own label, exactly like a point-in-time join never sees the future. The ordered target statistic for row i is:

TSi = ( Σ yj over earlier rows j<i with same category + a · prior ) / ( count of those rows + a )

The a · prior term (a smoothing prior, usually the global target mean, with strength a) keeps the estimate sane when the prefix is empty or tiny — the first time a category appears, its TS is just the prior. Slide the pointer below: the greedy encoding always averages the whole category and swallows the current row's own label (circled), while the ordered encoding only sees the shaded prefix.

Because a single permutation gives noisy, high-variance estimates for early rows, CatBoost draws several permutations and averages — reshuffle the widget above to see the ordered value move with the order. This is the whole trick, applied in two places.

Ordered boosting — the same trick on the gradients

CatBoost applies "only the past" to the boosting loop too. In ordered boosting it keeps a set of supporting models where model Mi is trained on the first i rows of the permutation only. To get the gradient for row i+1, it uses Mi — a model that has never seen row i+1. The residual is therefore an honest out-of-sample residual, not the optimistic in-sample one that classic boosting (L013/L014) uses. That removes the prediction-shift bias at the cost of maintaining several models, so CatBoost uses boosting_type="Ordered" by default on small data and falls back to fast "Plain" boosting on large data.

Oblivious (symmetric) trees

CatBoost's base learner is unusual: an oblivious tree uses the same split (feature + threshold) across an entire level. A depth-6 oblivious tree is really a decision table with 26 = 64 leaves indexed by 6 yes/no tests. Each individual tree is weaker and less flexible than XGBoost's free-form trees, but they are strongly regularized (harder to overfit) and give extremely fast, branch-free inference — you compute 6 bits and index a table. It is a deliberate bias-for-variance trade, in the same spirit as the shrinkage and subsampling knobs from L013–L015.

On our data

First, the leak, measured directly. On a pure-noise high-cardinality category (700 categories, target independent of the category, so the honest signal is zero), we treat each encoding as a one-feature score for y and read its ROC-AUC. A leakage-free feature should score ≈ 0.5:

Encoding of a noise categoryAUC(encoding, y)Verdict
Greedy TS (own row included)0.845spurious signal — leak
Ordered TS (prefix only, mean of 20 perms)0.493≈ 0.5 — no leak

On the small, categorical-rich credit_g CatBoost's ordered TS clearly help — 0.896, the best single-table number we've seen on it (past the tuned XGBoost 0.896 tie from L014 and Random Forest 0.901, and ahead of both boosters' defaults). On big adult, where categories are common and label leakage is diluted, all three are a near-tie — no free lunch.

Honest note on ordered boosting. Toggling just the boosting scheme on credit_g ("Ordered" 0.889 vs "Plain" 0.896, same 400 trees) shows ordered boosting is not a guaranteed accuracy win on a single dataset — it is a robustness/anti-overfitting device, and it is slower. Most of CatBoost's edge here comes from the ordered target statistics (native categoricals), not from ordered boosting. Report both honestly.

Thesis bridge. Relational databases are made of high-cardinality categorical keys — user_id, product_id, store_id foreign keys that are the entities. Ordered target statistics are the strongest leakage-free single-table way to turn "who/what" identity into a feature without ever building a graph. That makes CatBoost a direct baseline the relational thesis must beat: "we beat XGBoost" is not enough if you never tried CatBoost's native categorical handling on the flattened join. And note the deeper symmetry — CatBoost's random permutation is the same "only use the past" discipline that Relational Deep Learning must enforce with timestamps on the entity graph (Y3–Y4). Same leak, same cure, different data structure.

Where CatBoost breaks (know this too)

  • Oblivious trees are less expressive per tree; on problems dominated by complex numeric interactions (few categoricals), free-form XGBoost/LightGBM trees can edge it — see the adult near-tie.
  • Ordered boosting is slower and memory-heavier than Plain/LightGBM; the accuracy gain is dataset-dependent and sometimes zero (our credit_g toggle).
  • Careful manual target encoding inside a proper CV pipeline (L009) recovers much of the categorical advantage — CatBoost automates the discipline, it doesn't hold secret math.
  • Still a single-table model: ordered TS encodes one categorical column against the label; it never learns multi-hop structure across joins the way a GNN on the entity graph will.
