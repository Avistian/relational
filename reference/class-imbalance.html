Class Imbalance

Reference · print-friendly cheat sheet

The two rules

  1. Don't trust accuracy. On rare positives it just reports the majority rate.
  2. Resample only inside the CV fold (use imblearn.pipeline). Reweighting can't leak; resampling can.

Metrics (pick for the rare class)

MetricAnswersReach for it when
Recall / sensitivityof real positives, how many caughtmissing a positive is costly
Precisionof positive calls, how many rightfalse alarms costly
F1harmonic mean of the twoneed one balanced number
PR-AUC (avg precision)precision across thresholdsdefault for rare positives
Balanced accuracymean recall over classesaccuracy-like single number
ROC-AUCranking qualityfine, but optimistic under imbalance

Verified same-model contrast: ROC-AUC 0.914 vs PR-AUC 0.747. ROC flatters; PR tells the truth.

Handling menu (first to last)

MethodHowLeak risk
Class weights (cost-sensitive)class_weight="balanced"; XGB scale_pos_weightnone — no rows added
Threshold tuningmove decision threshold; choose on validationnone if chosen off test
ResamplingSMOTE / RandomUnderSampler, inside imblearn Pipelinehigh if done before CV

Verified: class_weight="balanced" recall 0.55 → 0.83, but PR-AUC 0.756 → 0.734 — reweighting shifts the operating point, it does not add ranking signal.

The resampling leak

# WRONG — SMOTE sees all rows, then CV grades on synthetic near-copies
X_res, y_res = SMOTE().fit_resample(X, y)
cross_val_score(clf, X_res, y_res, cv=cv, scoring="f1")     # 0.887 (fiction)

# RIGHT — imblearn Pipeline: resamples TRAIN folds only, predicts on natural imbalance
from imblearn.pipeline import make_pipeline
from imblearn.over_sampling import SMOTE
pipe = make_pipeline(SMOTE(random_state=0), clf)
cross_val_score(pipe, X, y, cv=cv, scoring="f1")            # 0.479 (honest)

Why imblearn.pipeline: SMOTE rewrites y, which a plain sklearn Pipeline can't host; imblearn also skips resamplers at predict time so test folds keep natural imbalance.

Relational / thesis notes

  • RelBench tasks (churn, fraud, conversion, link prediction) are rare-positive → score with PR-AUC / AUROC, never accuracy.
  • Across a temporal split, prefer scale_pos_weight / class_weight; SMOTE-before-split is resampling leak + time leak.
  • SMOTE on an entity graph is ill-defined (interpolating two customers + their orders breaks structure); RDL keeps real entities and carries cost in the loss.

Decision flow

Imbalanced target?
 ├─ Metric: drop accuracy → PR-AUC (+ recall at your cost point)
 ├─ First fix: class_weight / scale_pos_weight   (leak-free)
 ├─ Still need more recall? tune threshold on VALIDATION
 └─ Resample? only inside imblearn Pipeline, never before CV/temporal split
