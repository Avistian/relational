Design Matrix & Leakage

Reference · print-friendly cheat sheet

Contract

X ∈ ℝn×d   — n rows (examples), d columns (features)
y ∈ ℝn      — one label per row, same row order as X

Every sklearn / XGBoost / tabular neural API expects this pair. Multi-table SQL must be reduced to it upstream.

Leakage red flags in joins

PatternVerdictFix
AVG(amount) over all orders, no time cut Leaks Filter order_date ≤ t per prediction time t
COUNT(*) including events after label time Leaks Point-in-time (PIT) aggregate at t
Label column (or derivative) in features Leaks Drop target and post-outcome columns
Train/test split after global normalization Leaks Fit scaler on train only; transform val/test
COUNT orders in 30d window ending at t Safe — (if t is prediction time)
Static attributes known at signup (country, cohort) Safe — (verify not updated retroactively)

Sanity question

Before training: Could I compute this feature at prediction time using only information that existed then?
