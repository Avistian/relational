Pipelines & Preprocessing

Lesson 005 · Phase 1 · ~40 min

Lesson 004 built the discipline that makes a cross-validation number honest: keep groups whole, and never let the data that tuned a model also report its score. But there is one more way the future leaks into the past, and it is the most common silent bug in a "strong baseline" — preprocessing fit on data the model should not have seen.

This lesson's single skill: wrap every transformation — imputation, scaling, encoding, feature selection — inside an sklearn Pipeline so it is refit inside each fold, on training rows only. Get this right and your CV estimate finally measures what you claim it measures. Get it wrong and you have rebuilt the Lesson 002 leak through the back door.

The leak you cannot see in the metric

Recall Lesson 003's split hygiene table: "Fit scaler on all of X before split." Why is that wrong? A StandardScaler subtracts the mean and divides by the standard deviation. If you compute that mean and std over the whole dataset, the numbers carry information from the test rows into every training row. The same is true of:

The reason this bug is so persistent is that nothing warns you. The code runs, the score prints, and the number is simply too good. You only discover it when the model underperforms in production — or when a reviewer asks where your scaler was fit.

Under "fit on all data" every test square is ringed: the transformer peeked at all of them once, before the split. Under "Pipeline" no test square is ever ringed — the transformer is refit fold by fold, on train only.

How big is the lie? A measured demo

Theory is cheap; let's measure. Take pure noise — 200 rows, 2000 random features, a random binary label with no real relationship. Honest accuracy must be about 0.5. Now select the 20 features most correlated with y — but do it before cross-validation, on all the data — then CV a logistic regression on those 20 columns: 

rng = np.random.RandomState(0)
X = rng.randn(200, 2000)        # pure noise
y = rng.randint(0, 2, 200)      # random labels

# WRONG: select features on ALL data, then cross-validate
X_sel = SelectKBest(f_classif, k=20).fit_transform(X, y)
wrong = cross_val_score(LogisticRegression(max_iter=1000), X_sel, y, cv=cv).mean()

# RIGHT: selection lives inside the pipeline, refit each fold
pipe = make_pipeline(SelectKBest(f_classif, k=20),
                     LogisticRegression(max_iter=1000))
right = cross_val_score(pipe, X, y, cv=cv).mean()

Verified (sklearn 1.9): the leak invents 28 points of accuracy out of literally nothing — there is no signal in the data. The selector, having seen all labels, picks the 20 columns that happen to correlate by chance; those columns then "predict" in every fold because the choice already used the test labels. The Pipeline version refits the selector per fold and collapses back to chance.

Not every step leaks this dramatically. A StandardScaler on a real, stable dataset often shifts the score by a tiny amount, because per-column means barely move when you drop one fold. That is exactly the trap: the leak is sometimes invisible in the metric, so you cannot decide step-by-step whether to bother. The rule has to be unconditional — everything with a fit goes inside the Pipeline — precisely because you cannot eyeball which transforms are safe.

The tool: Pipeline

A Pipeline chains transformers and a final estimator into one object with one fit and one predict. When you hand it to cross_val_score or GridSearchCV, every fit call refits the whole chain on that fold's training rows, then applies it to the held-out rows. The leak becomes structurally impossible. 

from sklearn.pipeline import Pipeline, make_pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.impute import SimpleImputer
from sklearn.linear_model import LogisticRegression

# Explicit names — needed later for hyperparameter tuning
pipe = Pipeline([
    ("impute", SimpleImputer(strategy="median")),
    ("scale",  StandardScaler()),
    ("clf",    LogisticRegression(max_iter=1000)),
])

# Or let sklearn name the steps for you
pipe = make_pipeline(SimpleImputer(strategy="median"),
                     StandardScaler(),
                     LogisticRegression(max_iter=1000))

scores = cross_val_score(pipe, X, y, cv=5)   # each fold refits the whole chain

make_pipeline auto-names steps in lowercase (simpleimputer, standardscaler, logisticregression); Pipeline lets you name them. Use explicit names when you plan to tune — the names become the prefix in the parameter grid.

Mixed columns: ColumnTransformer

Real tables mix numeric and categorical columns that need different treatment: impute the median and scale the numbers; impute the mode and one-hot the categories. ColumnTransformer routes each column group to its own sub-pipeline and concatenates the results — all still refit per fold. 

from sklearn.compose import ColumnTransformer, make_column_selector
from sklearn.preprocessing import OneHotEncoder

numeric = Pipeline([
    ("impute", SimpleImputer(strategy="median")),
    ("scale",  StandardScaler()),
])
categorical = Pipeline([
    ("impute", SimpleImputer(strategy="most_frequent")),
    ("onehot", OneHotEncoder(handle_unknown="ignore", sparse_output=False)),
])

pre = ColumnTransformer([
    ("num", numeric,     make_column_selector(dtype_include="number")),
    ("cat", categorical, make_column_selector(dtype_include=object)),
])

model = Pipeline([("pre", pre), ("clf", LogisticRegression(max_iter=1000))])

Two settings here are not optional, and both come from verified runs:

  1. handle_unknown="ignore" on the encoder. A category that appears only in a test fold would otherwise raise at transform time and crash your CV. "Ignore" encodes the unseen category as all-zeros — the only sane choice once preprocessing is correctly refit per fold (new categories will appear).
  2. sparse_output=False if you call .set_output(transform="pandas"). Verified on sklearn 1.9: a pandas-output OneHotEncoder with the default sparse matrix raises "Pandas output does not support sparse data." Set it to dense, or keep numpy output.
Readable feature names. Call pre.set_output(transform="pandas") and the transformed columns come back named num__a, cat__cat_x … — the <step>__<feature> convention. Verified: invaluable when you later inspect coefficients or feature importances and need to know which engineered column is which.

Tuning the whole pipeline

Because the pipeline is one estimator, GridSearchCV tunes preprocessing and model together, addressing each parameter by <stepname>__<param> — the double underscore. This composes directly with the nested CV from Lesson 004: 

from sklearn.model_selection import GridSearchCV

param_grid = {
    "pre__num__impute__strategy": ["median", "mean"],
    "clf__C": [0.1, 1.0, 10.0],
}
search = GridSearchCV(model, param_grid, cv=5)
search.fit(X_train, y_train)

Read pre__num__impute__strategy left to right: the pre step → its num sub-pipeline → the impute step → its strategy argument. Every candidate is evaluated with preprocessing refit per inner fold — there is no path left for leakage.

Bridge to the thesis

This is not a detour from relational deep learning — it is the spine of a fair baseline. When you claim an RDL model beats a tuned GBDT, the GBDT's number must come from a pipeline where imputation, encoding, and any feature selection were refit on each training fold. A reviewer who finds a scaler fit on the full table will discard the comparison, and they will be right to.

There is a deeper echo, too. The aggregate features you sketched in Lesson 002 — "count of orders before time t" — are preprocessing. Relational deep learning's whole pitch is to learn those aggregations inside the model instead of hand-fitting them outside it. The Pipeline discipline you build here is the manual version of what an end-to-end relational model promises to automate. To argue the automated version is undervalued, you must first be flawless at the manual one.

Four mistakes to catch

  1. Scaler/imputer fit before the split. Test stats leak in. Fix: put it in the Pipeline.
  2. Feature selection before CV. The worst offender — test labels pick features. Fix: SelectKBest as a pipeline step.
  3. OneHotEncoder without handle_unknown="ignore". Crashes on a test-only category. Fix: ignore unknowns.
  4. Tuning preprocessing by hand outside CV. Silent leakage. Fix: put params in the grid with step__param.

Lab — turn a leak into an honest number

Exit ticket. In Python (notebook or script):

  1. Build a small mixed-type DataFrame: a couple of numeric columns (inject some NaNs) and one categorical column. Any classification label.
  2. Leak it on purpose. Fit a StandardScaler (or SelectKBest) on the whole X, then run cross_val_score. Record the mean.
  3. Fix it. Put the same preprocessing in a Pipeline (use ColumnTransformer for the mixed columns) and cross_val_score again. Record the mean.
  4. Confirm the leaked number is ≥ the pipeline number, and explain in one sentence which statistic leaked.
  5. Bonus: wrap the pipeline in GridSearchCV tuning one preprocessing param (e.g. impute__strategy) and one model param via step__param names.
