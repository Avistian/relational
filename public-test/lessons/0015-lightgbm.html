LightGBM

Lesson 015 · Phase 1 · ~40 min

Lesson 014 gave you XGBoost's exact machinery: the regularized leaf weight w* = −G/(H+λ), the structure-score split gain, and a first glimpse of histogram split finding. LightGBM (Ke et al., NeurIPS 2017) keeps that same gain math and re-engineers everything around it to be fast and scalable when rows and features explode. This lesson's single skill: explain the four tricks that make LightGBM fast — histograms, leaf-wise growth, GOSS, and EFB — and know that num_leaves (not max_depth) is now the central complexity knob, then confirm it against XGBoost on a real task.

One sentence. LightGBM is XGBoost's gradient/hessian gain formula with four speed-ups bolted on — bin features into histograms, grow trees leaf-wise instead of level-wise, throw away most small-gradient rows (GOSS), and merge sparse features that never co-occur (EFB).

Same gain, different growth: leaf-wise vs level-wise

XGBoost (by default) grows a tree level-wise: it splits every node at the current depth before going deeper, producing balanced trees. LightGBM grows leaf-wise (best-first): at each step it splits the one frontier leaf whose split would reduce the loss most — the leaf with max delta loss. Holding the number of leaves fixed, leaf-wise reaches lower training loss because it never wastes a split on a low-gain node.

The cost: leaf-wise trees get deep and unbalanced, which overfits on small data. That is why LightGBM's primary complexity knob is num_leaves (how many leaves the tree may have), with max_depth as an optional safety cap — a real shift from XGBoost, where max_depth was the main lever.

GOSS — keep the rows that still hurt

To score a split, XGBoost scans every row to sum gradients and hessians. LightGBM's Gradient-based One-Side Sampling asks: do we really need them all? A row with a large gradient is one the model still gets badly wrong — it carries most of the information about where to split. A row with a tiny gradient is already well-fit. So GOSS:

  • keeps the top a·n rows by |gradient| (the under-trained, informative ones),
  • randomly draws a further b·n rows from the small-gradient remainder,
  • and amplifies those sampled rows by (1−a)/b when summing, so the overall gradient statistics stay unbiased.
amplify small-gradient sample by (1 − a) / b

The paper proves this estimates the information gain accurately with far fewer rows — the source of much of the "up to 20× faster than conventional GBDT" headline (Ke et al. 2017, §3). It is a smarter cousin of the row subsampling from L012: instead of sampling uniformly, sample by how wrong the model still is.

EFB — bundle features that never co-occur

High-dimensional data (especially one-hot encodings) is usually sparse: many features are almost never nonzero at the same time. Exclusive Feature Bundling packs such mutually exclusive features into a single bundled feature — a one-hot block of 1,000 columns that only ever has one "hot" entry becomes, effectively, one column. Histogram building then costs O(#data × #bundles) instead of O(#data × #features). Finding the optimal bundling is NP-hard (it reduces to graph coloring), so LightGBM uses a greedy approximation (Ke et al. 2017, §4).

Histograms and the subtraction trick

Both XGBoost's hist mode and LightGBM bucket each feature's values into ~255 discrete bins, so a split scan costs O(#bins) instead of O(#rows) — the candidate-threshold idea previewed in L014. LightGBM adds a neat accountant's trick: a node's two children partition its rows, so childright histogram = parent histogram − childleft histogram. Build the histogram for the smaller child, subtract to get the sibling for free. These are engineering wins, not new statistics — the gain formula underneath is still L014's.

The "20× faster" myth, measured honestly

LightGBM's headline was "up to 20× faster than conventional GBDT" — and the operative words are conventional GBDT. The 2017 baseline was a pre-histogram implementation that scans every row for every split (like scikit-learn's GradientBoostingClassifier). Modern XGBoost's hist mode adopted the same histogram idea, so today the two are roughly on par. Timing a single 100-tree fit on a 50k × 50 synthetic set makes this concrete:

ImplementationFit timevs conventional
sklearn GBDT (conventional, pre-histogram)64.3 s1×
XGBoost (hist)0.58 s≈110×
LightGBM (default gbdt)0.60 s≈107×
LightGBM (boosting_type="goss")0.42 s≈152×

Two honest takeaways. First, the ~100× speed-up is real but historical: it is against pre-histogram GBDT, not against a modern XGBoost that already learned the trick — exactly the "untuned XGBoost ≈ 2016-era sklearn" lesson from L014, one layer up. Second, GOSS is opt-in: LightGBM's default boosting_type is gbdt, so you only get the extra ~30% here by asking for "goss". LightGBM's durable edges over XGBoost are elsewhere — native categorical splits, lower memory, and better scaling on sparse, high-cardinality data (where EFB pays off).

Thesis bridge. LightGBM is the scale member of the GBDT baseline family: when a relational join is flattened into millions of rows and thousands of sparse, one-hot columns, LightGBM (native categoricals + EFB + low memory) is often the single-table baseline that actually fits in RAM and finishes. "We beat XGBoost but never tried a tuned LightGBM on the full-size join" is not an honest win (Grinsztajn 2022; the 2026 fair-comparison stack still lists a tuned GBDT).

Where LightGBM breaks (know this too)

  • Leaf-wise growth overfits small/noisy data unless num_leaves (and often min_child_samples) are tuned down — see the sweep above.
  • The default num_leaves=31 is not a safe universal default; it interacts with max_depth in ways that surprise people migrating from XGBoost.
  • On tiny data the speed advantage is irrelevant and a Random Forest or tuned XGBoost can tie it (L012/L014).
  • Still a single-table model: faster, but it never sees relational structure you didn't hand-engineer across joins.
