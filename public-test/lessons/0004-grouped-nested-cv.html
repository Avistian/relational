Grouped & Nested Cross-Validation

Lesson 004 · Phase 1 · ~40 min

Lesson 003 ended on a warning: standard K-fold assumes rows are i.i.d. — independent draws from one distribution. You already use grouped CV in your pipelines, so this lesson is not about convincing you that groups matter. It is about making two failure modes precise, naming the exact sklearn tools, and adding the one technique that separates a tuned number from an honest tuned number: nested CV.

Both failures share a root cause from Lesson 002: information that should be unseen at scoring time bleeds in. There, it bled through features. Here, it bleeds through rows (the same group in train and test) and through the tuning procedure itself (the data that picked the hyperparameters also reports the score).

Failure 1 — the i.i.d. assumption breaks

K-fold shuffles rows and deals them into folds as if each row were an independent example. Real relational data violates this constantly, because one entity produces many rows:

Entity (group)Many rows because…Task
Customermany orders / sessionschurn, LTV
Patientrepeated visits / labsdiagnosis, readmission
Storeone row per daydemand forecasting
User–itemrepeated interactionsrecommendation

When the same customer appears in both train and test, the model can learn group-specific shortcuts — “customer #5012 churns” — instead of a generalizable pattern. The test fold then rewards memorization. The scikit-learn guide (§3.1.2.4) states the fix plainly:

scikit-learn: “GroupKFold is a variation of K-fold which ensures that the same group is not represented in both testing and training sets… if the model is flexible enough to learn from highly person-specific features it could fail to generalize to new subjects. GroupKFold makes it possible to detect this kind of overfitting.”

See the leak

Each row below is a group (a customer); each square is one of their samples (orders). Color = assigned fold. Toggle the modes. Under random K-fold, a group's squares scatter into different folds — that group is in both train and test. Under GroupKFold, every group is a single color: atomic, never split.

The tool: GroupKFold

You pass a groups array — one group id per row — and sklearn keeps every group whole: 

from sklearn.model_selection import GroupKFold, cross_val_score

gkf = GroupKFold(n_splits=5)
scores = cross_val_score(clf, X, y, groups=customer_id, cv=gkf)
print(f"{scores.mean():.3f} ± {scores.std():.3f}")

Three things that bite people:

  1. You must pass groups. Calling gkf.split(X, y) without it raises “The groups parameter should not be None.” train_test_split spoils you — group splitters never assume.
  2. Folds are not equal-sized. Groups have different row counts, so folds differ in size. That is correct behavior, not a bug.
  3. You need at least n_splits distinct groups. Five folds need five or more groups, or the split is impossible.

When you also need class balance: StratifiedGroupKFold

Plain GroupKFold balances the number of groups per fold, not the class ratio. On imbalanced churn that can hand you a fold with almost no positives — the Lesson 003 problem, now constrained by groups. StratifiedGroupKFold preserves the class percentages and keeps groups non-overlapping: 

from sklearn.model_selection import StratifiedGroupKFold

sgkf = StratifiedGroupKFold(n_splits=5, shuffle=True, random_state=42)
scores = cross_val_score(clf, X, y, groups=customer_id, cv=sgkf)

Per the API docs, when groups are few-but-large, perfect stratification is impossible and it degrades gracefully toward GroupKFold — groups win the tie, because group leakage is the more dangerous error.

Failure 2 — tuning and scoring on the same data

Lesson 003 taught: never tune on the test set. But there is a subtler leak. Suppose you run a 5-fold CV grid search, pick the hyperparameters with the best CV score, and then report that same best CV score as your performance estimate. You have used the validation folds twice — once to choose, once to report — so the number is optimistically biased. The estimate is contaminated by the selection.

Cawley & Talbot (JMLR 2010): a model-selection criterion with non-negligible variance can be over-fit just like model parameters; the resulting bias is “often of comparable magnitude to differences in performance between learning algorithms.” In plain terms: the gap you brag about may be smaller than the bias your protocol introduced.

This is exactly the trap your thesis cannot fall into. If you claim a relational model beats a tuned GBDT, a skeptic's first question is: was the GBDT's reported number chosen by the same data that scored it? If yes, the comparison is dead on arrival.

The tool: nested cross-validation

Nested CV uses two loops. The inner loop searches hyperparameters. The outer loop scores the whole search-and-fit procedure on data the inner loop never saw. Each outer fold's hyperparameters are chosen fresh, inside that fold only.

OUTER fold 1 ──▶ [ INNER CV: grid search → best params ] ──▶ fit ──▶ score on outer test 1 OUTER fold 2 ──▶ [ INNER CV: grid search → best params ] ──▶ fit ──▶ score on outer test 2 OUTER fold 3 ──▶ [ INNER CV: grid search → best params ] ──▶ fit ──▶ score on outer test 3 │ unbiased estimate = mean ± std of outer ◀──┘

In sklearn the pattern is a GridSearchCV (the inner loop) handed to cross_val_score (the outer loop): 

from sklearn.model_selection import GridSearchCV, cross_val_score, KFold

inner = KFold(n_splits=5, shuffle=True, random_state=1)
outer = KFold(n_splits=5, shuffle=True, random_state=2)

search = GridSearchCV(clf, param_grid, cv=inner)
nested_scores = cross_val_score(search, X, y, cv=outer)
print(f"Nested estimate: {nested_scores.mean():.3f} ± {nested_scores.std():.3f}")

The number you report is nested_scores.mean(). The hyperparameters change across outer folds — and that is fine: nested CV evaluates the procedure, not one frozen configuration. You pick the final config separately, by running the inner search once on all the data after you have your honest estimate.

Two CVs, two questions

LoopAnswersOutput you keep
Inner“Which hyperparameters?”best params (per outer fold)
Outer“How well does the whole procedure generalize?”mean ± std — the headline number

Combining both: grouped + nested

Your relational baselines often need both — group isolation and an unbiased tuned estimate. Use group splitters in both loops and route the groups array through. There are two ways, and the first one trips everyone up.

Metadata routing (sklearn ≥ 1.4) lets cross_val_score forward groups to the inner search — but you must opt in first, or it raises “The groups parameter should not be None.”: 

import sklearn
from sklearn.model_selection import (
    GridSearchCV, cross_val_score, GroupKFold)

sklearn.set_config(enable_metadata_routing=True)   # easy to forget!

search = GridSearchCV(clf, param_grid, cv=GroupKFold(5))
nested = cross_val_score(
    search, X, y,
    cv=GroupKFold(5),
    params={"groups": customer_id},
)

Portable manual loop — works on any sklearn version and makes the plumbing explicit. Write the outer loop yourself and pass groups to both split and fit: 

import numpy as np
scores = []
for tr, te in GroupKFold(5).split(X, y, groups=customer_id):
    search = GridSearchCV(clf, param_grid, cv=GroupKFold(5))
    search.fit(X[tr], y[tr], groups=customer_id[tr])   # inner loop
    scores.append(search.score(X[te], y[te]))          # outer score
print(f"{np.mean(scores):.3f} ± {np.std(scores):.3f}")

The plumbing of groups through nested loops is the single most error-prone part of an honest baseline — sliced into customer_id[tr] in the manual version, silently dropped if you forget set_config in the routing version. When in doubt, use the manual loop: you can see that no group crosses the boundary.

Bridge to the thesis

Grouping is the gateway drug to temporal splits. A customer is a group in space; a time cutoff is a “group” in time — you never let the future train the past. RelBench tasks demand time-respecting splits, which is Lesson 021. Everything you internalize here — the unit of independence is not the row — is the same instinct you will apply to time in Year 4. When you report that an RDL model beats a GBDT, both numbers must come from grouped, nested, time-correct protocols. Otherwise the undervaluation thesis rests on a measurement artifact.

Four mistakes to catch

  1. Row-level K-fold on grouped data. Same entity in train and test. Fix: GroupKFold.
  2. GroupKFold on imbalanced labels. A fold with no positives. Fix: StratifiedGroupKFold.
  3. Reporting the best grid-search CV score. Selection bias. Fix: nested CV; report the outer mean.
  4. Dropping groups in the outer loop. Groups leak across the nesting. Fix: route groups to both loops.

Check your understanding

Primary reading (today)

Read Cawley & Talbot, “On Over-fitting in Model Selection and Subsequent Selection Bias in Performance Evaluation” (JMLR 2010) — at minimum the abstract, §1 (introduction), and the discussion of nested / “double” cross-validation. It is the citation you will reach for whenever a reviewer doubts your baseline. Pair it with the sklearn grouped-data section (§3.1.2.4).

Lab — measure the optimism, then remove it

Exit ticket. In Python (notebook or script):

  1. Make grouped data: take any classification dataset and assign a synthetic group_id so each group owns several rows (e.g. 20 groups). Make the label partly group-determined so leakage actually helps the model cheat.
  2. Score a classifier two ways on the same data: plain KFold(5) vs GroupKFold(5) with groups=group_id. Print both means. The plain-KFold number should be optimistically higher — that gap is the leakage you would have shipped.
  3. Run a small GridSearchCV (e.g. 2–3 values of one hyperparameter) and print its best_score_. Then wrap it in cross_val_score as a nested CV and print the outer mean. Note that the nested estimate is lower (or equal) — that drop is the selection bias Cawley & Talbot warn about.
  4. Bonus: combine — GroupKFold in both nested loops, routing groups through.

Paste your four numbers (KFold mean, GroupKFold mean, grid best_score_, nested mean) or say “lab done.” Lesson 005 (pipelines & preprocessing) assumes you can write a leakage-safe split cold.
