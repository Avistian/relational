The Single-Table Assumption

Lesson 001 · Phase 1 · ~40 min

Your contrarian bet targets relational deep learning — but almost every tabular library, Kaggle notebook, and production pipeline starts from the same hidden contract: one row = one training example, columns = features. Before RDL can win, you need to see exactly what that contract costs.

What tabular ML actually eats

scikit-learn, XGBoost, and neural tabular models expect a matrix X and a vector y. Real companies store data in many linked tables — customers, orders, products, sessions. Someone must join, aggregate, and featurize those tables into one flat file. That work is feature engineering.

The contract is strict: row i of X must align with label y[i]. The learner never sees table names, foreign keys, or join graphs — only numbers in a rectangle. Every relational ML project therefore has two systems: the database (truth) and the design matrix (what the model sees). Lesson 002 makes that second system precise.

Fey et al. (ICML 2024): feature engineering on relational data is “slow, error prone and leads to suboptimal models” because no standard learner consumes multiple connected tables end-to-end.
Position: Relational Deep Learning

Worked example: e-commerce churn

Imagine a subscription retailer. The warehouse holds at least four tables: 

customers(customer_id, signup_date, country, plan_tier)
orders(order_id, customer_id, order_date, amount, category)
sessions(session_id, customer_id, start_time, pages_viewed)
support_tickets(ticket_id, customer_id, opened_at, resolved_at, topic)

The business question: will customer c cancel in the next 30 days? A data scientist’s notebook almost always begins with SQL like: 

SELECT c.customer_id,
       DATEDIFF('day', c.signup_date, :as_of_date) AS tenure,
       COUNT(o.order_id) AS order_count_90d,
       AVG(s.pages_viewed) AS avg_pages,
       COUNT(t.ticket_id) AS ticket_count
FROM customers c
LEFT JOIN orders o ON ... AND o.order_date BETWEEN ...
LEFT JOIN sessions s ON ...
LEFT JOIN support_tickets t ON ...
GROUP BY c.customer_id

Notice what happened. Four tables became one row per customer. Join keys disappeared. The graph of “who bought what, when, and after which session” collapsed into four hand-picked aggregates. That collapse is not neutral — it is a modeling choice baked into SQL that the gradient booster will never question.

Three ways the assumption breaks

  1. Lost structure. Primary–foreign key links encode who bought what, who knows whom, which event belongs to which user. Flattening collapses that graph into hand-picked aggregates (count_orders_30d, avg_session_length). Neighbors in the graph — products co-purchased, users in the same household — never appear unless someone engineers them.
  2. Leakage risk. A careless join can smuggle future information into features. Temporal relational data needs time-respecting splits — the REG in RDL marks what was known when. A feature that sums all orders, not just orders before the prediction date, can look predictive while cheating.
  3. Suboptimal signal. RelBench’s user study found GNNs on the full relational graph matched or beat an experienced data scientist’s manual features while cutting human effort by an order of magnitude (Robinson et al., NeurIPS 2024). That is the gap your thesis bets on — but only if evaluation is fair.

What relational deep learning proposes

RDL keeps the database shape. Fey et al. introduce a relational entity graph (REG): tables become node types, rows become nodes, foreign keys become edges. Message-passing neural networks (Year 3) can then learn over that graph instead of over a pre-flattened CSV.

This does not eliminate feature engineering overnight. You still need temporal splits, point-in-time labels, and honest baselines. Phase 1 exists so you can say, with evidence: “I beat a strong flat-table XGBoost on the same prediction task before I claim the graph helped.”

Common objections (and honest answers)

“We already ETL into a feature store — isn’t that enough?”
Feature stores version and serve aggregates; they do not remove the single-table contract. They industrialize the flattening. RDL asks whether the flattening step should be learned rather than hand-specified.
“Graphs are slow and hard to deploy.”
True for naive implementations. That is an engineering problem, not a reason to ignore lost structure. Your thesis lives at the frontier where graph models start winning on real benchmarks — RelBench, ContextGNN, RelGT — not in toy demos.
“GBDTs already win Kaggle tabular.”
On single-table competitions, yes — Grinsztajn et al. and TabArena confirm this under honest temporal splits. Your bet is that production data is rarely truly single-table, and that relational models close the gap once evaluation respects time.
