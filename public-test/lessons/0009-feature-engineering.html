Feature Engineering

Lesson 009 · Phase 1 · ~40 min

Lessons 005–008 taught you to evaluate a model honestly. This lesson is about giving the model something worth evaluating: features. A feature is a calculated field derived from raw columns, designed so the relationship the model needs to learn becomes one it can learn. This is the last concept before your Q1 checkpoint — and it is the exact skill the thesis is about replacing.

This lesson's single skill: engineer features that encode structure a model cannot synthesize on its own — and fit every data-dependent feature inside the pipeline so it never leaks.

Why models need help

You might assume a powerful model "figures out" whatever transformation it needs. Heaton (2016) tested this directly: he generated datasets built around specific expressions and measured whether neural nets, random forests, gradient boosting, and SVMs could synthesize each one from raw inputs. Simple single-column transforms (log, polynomial, power, root) were learned easily — but none of the four models could learn a ratio of differences, \( (a-b)/(c-d) \) (Heaton 2016). If a model can already synthesize a feature, engineering it is wasted effort; the value is in the features it cannot.

Verified (sklearn 1.9, target \(y=(a-b)/(c-d)\), 5-fold R²): a Ridge regressor scores R² 0.644 on the four raw columns and 0.998 once the ratio is engineered in. Even a strong HistGradientBoosting rises from 0.968 → 0.980 — it can locally approximate the smooth ratio, but still gets a free, leak-proof lift from being handed the exact structure. The cloud doesn't change; your coordinates do.

The high-value patterns

Ratios & differencesrate per unit, margin, changedivision/ratio-of-differences isn't synthesizable (Heaton)

Datetime is the cheapest win. A raw Unix timestamp hides everything useful inside one monotonic number. Split it into hour-of-day, day-of-week, and (for periodicity) \( \sin(2\pi\,h/24),\ \cos(2\pi\,h/24) \), and a cyclical signal appears:

The relational case — where the thesis lives

The most powerful features for relational data aren't in your table at all — they're aggregations over related rows in other tables: a user's transaction count, their average order value, days since last login, max session length. This is exactly the manual work a data scientist does before training a tabular model on a database.

Kanter & Veeramachaneni formalized and automated it as Deep Feature Synthesis: follow the foreign-key relationships to a base table, then stack aggregation primitives (SUM, MEAN, COUNT) and transform primitives (MONTH, HOUR) along that path — the "depth" is how many relationships you traverse (Kanter & Veeramachaneni 2015, the algorithm behind Featuretools). Their automated system beat 615 of 906 human teams across three competitions.

This is the thesis in one sentence. Manual relational feature engineering → Deep Feature Synthesis (automated, but you still pick primitives) → Relational Deep Learning (the GNN learns the aggregations end-to-end over the schema graph). RDL is trying to make this entire lesson obsolete. To prove it does, you must first be able to build the strong hand-engineered baseline it has to beat.

The leakage trap — non-negotiable

Any feature whose computation looks at the target or uses statistics of the data is a fit-bearing transform, and from Lesson 005 you know where those belong: inside the Pipeline, fit on training folds only. The classic offender is target/mean encoding — replacing a category with the mean target for that category. Computed on the full dataset it leaks the answer and inflates CV; it must be fit per-fold (or with out-of-fold smoothing). 

from sklearn.pipeline import Pipeline
from sklearn.preprocessing import FunctionTransformer

# Stateless engineering (ratios, datetime parts): safe anywhere, but keep it in the pipeline
ratio = FunctionTransformer(lambda X: np.c_[X, (X[:,0]-X[:,1])/(X[:,2]-X[:,3])])

# Data-dependent engineering (target/mean encoding, scaling, KBins) MUST be a pipeline step
pipe = Pipeline([
    ("engineer", ratio),
    ("encode",   TargetEncoder()),     # fit on train folds only — never the whole frame
    ("model",    HistGradientBoostingClassifier()),
])

Rule of thumb: a feature that uses only within-row arithmetic (a ratio of this row's columns, this row's hour) is stateless and cannot leak. A feature that uses across-row statistics (a category's mean target, a column's mean for imputation, a standardization scale) is fit-bearing and must live in the pipeline. For relational aggregations, the same discipline plus a time cutoff — only aggregate child rows that existed before the prediction time (your point-in-time rule from Lesson 002).

Four mistakes to catch

  1. Engineering features the model already learns. Logs/polynomials of one column rarely help trees. Fix: spend effort on ratios, interactions, and aggregations.
  2. Target encoding on the full dataset. Leaks the label. Fix: a pipeline step fit per fold.
  3. Aggregating across the time cutoff. Uses the future. Fix: point-in-time-correct windows only.
  4. Integer hour with a midnight cliff. 23 and 0 look far apart. Fix: sin/cos cyclical encoding.
